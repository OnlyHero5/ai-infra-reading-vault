# 训练主循环 · 源码走读

> 走读顺序：同步 `train.py` 的 bootstrap、显存交接、generate/train/update/eval 时序；再看 `train_async.py` 如何用预取把 rollout 和训练错开；最后看 periodic action 的边界规则。

---

## 1. sync train.py：启动与资源边界

### 1.1 logger、placement group、tracking

**问题与约束：** 训练入口要先建立日志和 Ray 资源，再初始化 tracking；如果资源还没分配就创建 actor 或记录训练状态，后续定位问题会很混乱。

**设计选择：** `train(args)` 的开头固定执行 `configure_logger()`、`create_placement_groups(args)`、`init_tracking(args)`。

**Explain：** 同步训练主循环的第一步是把“可观测性”和“GPU 资源拓扑”准备好。

来源：train.py L9-L13

**Code：**

```python
def train(args):
    configure_logger()
    pgs = create_placement_groups(args)
    init_tracking(args)
```

**代码逻辑：** logger 最先配置；Ray placement group 返回 `pgs` 字典；tracking 在资源创建之后初始化。

**为什么这样写：** 后面的 rollout manager、actor model、critic model 都依赖 placement group；tracking 也应记录完整运行，而不是在 actor 创建失败后才初始化。

**不变量与失败模式：** `pgs` 必须至少包含后续访问的 `"rollout"` 和训练相关资源；placement group 创建失败时不应继续创建模型 actor。

**Comment：** 这段和 [[06-PlacementGroup-00-MOC]] 连起来看，能理解 Slime 如何先规划 Ray 资源再启动角色。

### 1.2 RolloutManager 先于训练模型

**问题与约束：** 训练模型创建时可能需要知道 rollout manager 或 rollout 数量信息；RolloutManager 内部还会启动 SGLang engines。

**设计选择：** 先调用 `create_rollout_manager(args, pgs["rollout"])`，拿到 `rollout_manager` 和 `num_rollout_per_epoch`；再调用 `create_training_models(...)`。

**Explain：** Slime 的训练入口把 rollout 侧作为训练模型的外部依赖先建好。

来源：train.py L15-L20

**Code：**

```python
rollout_manager, num_rollout_per_epoch = create_rollout_manager(args, pgs["rollout"])

actor_model, critic_model = create_training_models(args, pgs, rollout_manager)
```

**代码逻辑：** rollout manager 使用 rollout placement group；训练模型使用完整 placement group 字典，并接收 rollout manager 作为参数。

**为什么这样写：** `num_rollout_per_epoch` 后续用于 save/eval 周期边界；训练模型也要能把权重同步到 rollout 侧。

**不变量与失败模式：** `create_rollout_manager` 必须在需要 SGLang engines 的路径完成初始化；如果 rollout manager 创建失败，训练模型不应继续启动。

**Comment：** 入口顺序体现了 Slime 的闭环：先有可生成数据的 rollout 侧，再创建会消费数据并更新权重的训练侧。

### 1.3 bootstrap 权重和 KV

**问题与约束：** colocate / offload 场景下，训练和 rollout 共享或争用 GPU 显存；首次 generate 前，rollout 侧必须有最新 actor 权重和可用 KV cache。

**设计选择：** 若 `offload_rollout`，先 onload weights；随后始终执行一次 `actor_model.update_weights()`；可选检查权重一致性；最后在 offload 场景 onload KV。

**Explain：** 这是训练开始前的“rollout 侧冷启动同步”。

来源：train.py L22-L32

**Code：**

```python
if args.offload_rollout:
    ray.get(rollout_manager.onload_weights.remote())

actor_model.update_weights()

if args.check_weight_update_equal:
    ray.get(rollout_manager.check_weights.remote(action="compare"))

if args.offload_rollout:
    ray.get(rollout_manager.onload_kv.remote())
```

**代码逻辑：** rollout 权重可能先被载回 GPU；actor 权重推送到 rollout；检查开关打开时调用 remote compare；KV cache 在权重同步后恢复。

**为什么这样写：** 权重必须先于生成使用；KV 和 CUDA graph 等运行态内存也要在 generate 前恢复，但权重同步本身不应依赖已经 onload KV。

**不变量与失败模式：** 首次 generate 前必须完成一次 `update_weights()`；如果 offload 场景漏掉 onload，SGLang engine 可能没有可用权重或 KV 内存。

**Comment：** 这段解释了为什么同步训练入口还没进主循环就已经发生一次权重同步。

### 1.4 eval-only 特例

**问题与约束：** 用户可能只想启动环境并做评测，不跑训练 rollout；此时 `num_rollout == 0`，主循环不会进入。

**设计选择：** 在主循环前专门判断 `args.num_rollout == 0 and args.eval_interval is not None`，直接触发 `rollout_manager.eval(rollout_id=0)`。

**Explain：** eval-only 不是训练循环的一个普通 step，而是主循环外的特殊出口。

来源：train.py L34-L36

**Code：**

```python
if args.num_rollout == 0 and args.eval_interval is not None:
    ray.get(rollout_manager.eval.remote(rollout_id=0))
```

**代码逻辑：** 只有 rollout 数为 0 且配置了 eval interval 时才触发一次 eval。

**为什么这样写：** 主循环以 `range(start_rollout_id, num_rollout)` 驱动；`num_rollout == 0` 时不会有任何循环内 eval 机会。

**不变量与失败模式：** eval-only 仍依赖前面的 rollout manager 和权重同步 bootstrap；如果没有 `eval_interval`，这个入口不会自动评测。

**Comment：** 这段是理解“只评测不训练”配置行为的关键。

### 1.5 offload_train 与 save 闭包

**问题与约束：** 每轮训练后可能需要清训练侧显存，也可能需要保存 actor、critic 和 rollout 数据源状态；这些操作受 critic-only 阶段、最后一轮和全局数据集影响。

**设计选择：** 在 `train` 内定义 `offload_train(actor_trains_this_step)` 和 `save(rollout_id)` 两个闭包，复用主循环上下文中的 actor、critic、rollout manager 和 args。

**Explain：** 这两个闭包把主循环的“周期性副作用”从 generate/train 主链里分离出来。

来源：train.py L38-L60

**Code：**

```python
def offload_train(actor_trains_this_step):
    if not args.offload_train:
        if not args.use_critic or actor_trains_this_step:
            actor_model.clear_memory()
        else:
            critic_model.clear_memory()

def save(rollout_id):
    actor_trains_this_step = (not args.use_critic) or rollout_id >= args.num_critic_only_steps
    if actor_trains_this_step:
        actor_model.save_model(
            rollout_id,
            force_sync=rollout_id == args.num_rollout - 1,
        )
    if args.use_critic:
        critic_model.save_model(
            rollout_id,
            force_sync=rollout_id == args.num_rollout - 1,
        )
    if args.rollout_global_dataset:
        ray.get(rollout_manager.save.remote(rollout_id))
```

**代码逻辑：** `offload_train` 在非自动 offload 时清理 actor 或 critic 显存；`save` 按 actor 是否训练、critic 是否启用和是否最后一轮决定保存对象与同步方式。

**为什么这样写：** critic-only 预热阶段不一定训练 actor；最后一轮 checkpoint 需要强制同步；rollout global dataset 还需要保存 rollout manager 侧状态。

**不变量与失败模式：** `actor_trains_this_step` 的判定必须与训练分支一致；否则可能保存未更新的 actor，或清掉下一步仍需保留的模型内存。

**Comment：** 保存和清显存看似辅助逻辑，但它们直接影响 colocate 训练能否稳定跑完整轮次。

## 2. sync train.py：主循环时序

### 2.1 循环前 eval

**问题与约束：** 有些训练希望在第一个 rollout 前先做一次 baseline eval；但这不应在恢复训练的每个 rollout 上重复发生。

**设计选择：** 主循环内只在 `rollout_id == 0` 且没有 `skip_eval_before_train` 时执行一次 eval。

**Explain：** 这是训练开始前的初始评测点。

来源：train.py L64-L65

**Code：**

```python
if args.eval_interval is not None and rollout_id == 0 and not args.skip_eval_before_train:
    ray.get(rollout_manager.eval.remote(rollout_id))
```

**代码逻辑：** 需要 eval interval 非空、当前 rollout 为 0、且未跳过训练前 eval，三个条件同时满足才触发。

**为什么这样写：** `start_rollout_id` 可能不是 0；恢复训练时不应把第一个恢复 step 当作全新训练的 baseline。

**不变量与失败模式：** 这不是周期 eval 的替代；循环末尾仍会通过 `should_run_periodic_action` 处理 eval interval。

**Comment：** 这段和 eval-only 特例不同：它发生在正常训练主循环内。

### 2.2 generate 后再 train

**问题与约束：** 同步模式下一轮训练应消费同一个 `rollout_id` 生成的数据；若 rollout 和训练显存 colocate，生成后还要把 rollout 侧 offload 掉。

**设计选择：** 每轮先同步等待 `rollout_manager.generate`，再可选 `rollout_manager.offload`，随后按 critic/actor 训练阶段调用异步训练并 `ray.get` 等待完成。

**Explain：** 同步主循环的核心是严格的 generate → train 顺序。

来源：train.py L67-L81

**Code：**

```python
rollout_data_ref = ray.get(rollout_manager.generate.remote(rollout_id))

if args.offload_rollout:
    ray.get(rollout_manager.offload.remote())

actor_trains_this_step = (not args.use_critic) or rollout_id >= args.num_critic_only_steps

if args.use_critic:
    value_refs = critic_model.async_train(rollout_id, rollout_data_ref)
    if actor_trains_this_step:
        ray.get(actor_model.async_train(rollout_id, rollout_data_ref, external_data=value_refs))
    else:
        ray.get(value_refs)
else:
    ray.get(actor_model.async_train(rollout_id, rollout_data_ref))
```

**代码逻辑：** `generate` 的结果作为训练输入；critic 启用时先启动 critic train 得到 value refs；actor 训练阶段把 value refs 作为 external data；critic-only 阶段只等待 critic。

**为什么这样写：** value 估计和 actor 更新可以通过 Ray 引用串接；同步主循环用 `ray.get` 保证当前 rollout 数据训练完成后才进入保存和权重更新。

**不变量与失败模式：** `rollout_data_ref` 必须与当前 `rollout_id` 对齐；critic-only 阶段不能误启动 actor，否则会破坏预热策略。

**Comment：** `async_train` 名字表示 Ray actor 方法异步返回引用，主循环仍通过 `ray.get` 保持 step 同步。

### 2.3 save / clear / update / eval 的尾部顺序

**问题与约束：** 一轮训练完成后，要保存、释放训练显存、恢复 rollout 权重/KV、推送新 actor 权重，并可能 eval；顺序错了会导致显存冲突或 eval 使用旧权重。

**设计选择：** 固定顺序：periodic save → `offload_train` → onload rollout weights → `actor_model.update_weights()` → onload rollout KV → periodic eval。

**Explain：** 这段是同步训练与 rollout serving 之间的显存和权重交接协议。

来源：train.py L83-L95

**Code：**

```python
if should_run_periodic_action(rollout_id, args.save_interval, num_rollout_per_epoch, args.num_rollout):
    save(rollout_id)

offload_train(actor_trains_this_step)
if args.offload_rollout:
    ray.get(rollout_manager.onload_weights.remote())
actor_model.update_weights()

if args.offload_rollout:
    ray.get(rollout_manager.onload_kv.remote())

if should_run_periodic_action(rollout_id, args.eval_interval, num_rollout_per_epoch):
    ray.get(rollout_manager.eval.remote(rollout_id))
```

**代码逻辑：** save 使用周期判断；训练侧内存随后释放；rollout 权重载回后接受 actor 权重更新；KV 恢复后才允许 eval。

**为什么这样写：** eval 应该看到训练后的最新 rollout 权重；colocate 场景下，rollout onload 前必须尽量释放训练侧显存。

**不变量与失败模式：** `update_weights()` 必须发生在下一轮 generate 前；如果 eval 放在 update 前，会评测旧策略。

**Comment：** 这段顺序可以作为排查 colocate 显存和权重不同步问题的检查表。

### 2.4 退出清理

**问题与约束：** 训练结束后 Ray rollout actor 可能仍持有 serving engine、router、数据源或后台资源；tracking 也需要正常结束。

**设计选择：** 主循环结束后调用 `rollout_manager.dispose`，再 `finish_tracking(args)`。

**Explain：** 清理顺序是先释放 rollout 侧，再结束记录。

来源：train.py L97-L98

**Code：**

```python
ray.get(rollout_manager.dispose.remote())
finish_tracking(args)
```

**代码逻辑：** 等待 rollout manager 远端 dispose 完成，然后关闭 tracking。

**为什么这样写：** 远端资源释放也可能产生日志和指标；tracking 应最后结束。

**不变量与失败模式：** 如果训练中途异常退出，这段不会自动执行，外部运行器需要负责 Ray 资源清理。

**Comment：** 正常训练路径的资源生命周期在这里闭合。

## 3. train_async.py：预取式异步训练

### 3.1 禁止 colocate

**问题与约束：** 异步预取会在训练当前 rollout 的同时提前启动下一轮 generate；colocate 需要严格 offload/onload 显存时序，两者冲突。

**设计选择：** `train_async.py` 开头直接断言 `not args.colocate`。

**Explain：** async train 只适合 rollout 与训练 GPU 分离的资源布局。

来源：train_async.py L10-L11

**Code：**

```python
def train(args):
    assert not args.colocate, "Colocation is not supported for async training."
```

**代码逻辑：** 进入函数第一步就检查 colocate；失败则直接中止。

**为什么这样写：** 如果允许 colocate，下一轮 generate 可能在当前训练还占用显存时启动，破坏同步版精心安排的 offload 顺序。

**不变量与失败模式：** async 入口假设训练和 rollout 资源可并行；colocate 必须使用同步主循环或其他明确支持 offload 的方案。

**Comment：** 这不是功能缺失，而是异步预取策略的资源前提。

### 3.2 async bootstrap

**问题与约束：** 异步训练仍要建立 Ray 资源、rollout manager、训练模型，并把初始 actor 权重推到 rollout；但不需要同步版的 rollout onload/offload 分支。

**设计选择：** 保留 logger、placement group、tracking、rollout manager、training models 和首次 `update_weights()`；去掉 `offload_rollout` 相关操作。

**Explain：** async bootstrap 是同步 bootstrap 的简化版。

来源：train_async.py L12-L28

**Code：**

```python
configure_logger()
pgs = create_placement_groups(args)
init_tracking(args)

rollout_manager, num_rollout_per_epoch = create_rollout_manager(args, pgs["rollout"])

actor_model, critic_model = create_training_models(args, pgs, rollout_manager)

actor_model.update_weights()

if args.check_weight_update_equal:
    ray.get(rollout_manager.check_weights.remote(action="compare"))
```

**代码逻辑：** 初始化顺序和同步版一致；首次权重同步仍然执行；只保留可选权重比较。

**为什么这样写：** 异步只改变 generate/train 重叠方式，不改变初始权重一致性的要求。

**不变量与失败模式：** 首次 `rollout_manager.generate` 前必须已经 `update_weights()`；否则预取的第一批样本可能来自未同步权重。

**Comment：** 和同步版相比，缺少 onload/offload 正是因为 async 禁止 colocate。

### 3.3 预取主循环

**问题与约束：** 要让 rollout generation 与训练重叠，但训练第 `rollout_id` 步仍必须消费对应的 rollout 数据。

**设计选择：** 循环前先启动 `start_rollout_id` 的 generate；每轮先 `ray.get` 上一轮启动的 future，再立即启动下一轮 generate，然后训练当前数据。

**Explain：** async 主循环通过“一步 ahead 的 future”实现预取。

来源：train_async.py L31-L49

**Code：**

```python
rollout_data_next_future = rollout_manager.generate.remote(args.start_rollout_id)
for rollout_id in range(args.start_rollout_id, args.num_rollout):
    if rollout_data_next_future is not None:
        rollout_data_curr_ref = ray.get(rollout_data_next_future)

    if rollout_id + 1 < args.num_rollout:
        rollout_data_next_future = rollout_manager.generate.remote(rollout_id + 1)

    if args.use_critic:
        actor_trains_this_step = rollout_id >= args.num_critic_only_steps
        value_refs = critic_model.async_train(rollout_id, rollout_data_curr_ref)
        if actor_trains_this_step:
            ray.get(actor_model.async_train(rollout_id, rollout_data_curr_ref, external_data=value_refs))
        else:
            ray.get(value_refs)
    else:
        ray.get(actor_model.async_train(rollout_id, rollout_data_curr_ref))
```

**代码逻辑：** `rollout_data_next_future` 在当前轮开头变成 `rollout_data_curr_ref`；下一轮 generate 在训练当前数据前启动；critic/actor 训练逻辑与同步版基本一致。

**为什么这样写：** generate 往往是长耗时步骤，提前启动下一轮可以隐藏一部分 rollout 延迟；训练数据对齐仍由 future 的 rollout_id 保证。

**不变量与失败模式：** 在每次训练前 `rollout_data_curr_ref` 必须已由对应 future 得到；若 generate 抛错，会在 `ray.get` 当前轮数据时暴露。

**Comment：** 这是普通 async train，不是 fully-async rollout；fully-async 另见 [[14-Alt-Rollout-02-源码走读]]。

### 3.4 保存、稀疏权重更新与 eval

**问题与约束：** async 模式下下一轮 generate 可能正在跑；如果此时直接更新 rollout 权重，就可能在生成中途改变权重。

**设计选择：** save 仍按 periodic action；当 `(rollout_id + 1) % update_weights_interval == 0` 时，先 `ray.get` 未完成的 next future，再置空 future，最后 `actor_model.update_weights()`。

**Explain：** async 的权重更新必须先收束正在进行的 generate。

来源：train_async.py L51-L72

**Code：**

```python
if should_run_periodic_action(rollout_id, args.save_interval, num_rollout_per_epoch, args.num_rollout):
    if (not args.use_critic) or rollout_id >= args.num_critic_only_steps:
        actor_model.save_model(
            rollout_id,
            force_sync=rollout_id == args.num_rollout - 1,
        )
    if args.use_critic:
        critic_model.save_model(
            rollout_id,
            force_sync=rollout_id == args.num_rollout - 1,
        )
    if args.rollout_global_dataset:
        ray.get(rollout_manager.save.remote(rollout_id))

if (rollout_id + 1) % args.update_weights_interval == 0:
    rollout_data_curr_ref = ray.get(x) if (x := rollout_data_next_future) is not None else None
    rollout_data_next_future = None
    actor_model.update_weights()

if should_run_periodic_action(rollout_id, args.eval_interval, num_rollout_per_epoch):
    ray.get(rollout_manager.eval.remote(rollout_id))
```

**代码逻辑：** 保存逻辑内联在 async 主循环；到达更新间隔时先等待已预取的 generate 完成，再清空 future 并更新权重；eval 仍按 periodic action。

**为什么这样写：** 预取提升吞吐，但权重更新是全局一致性边界；必须避免同一个 generate 请求前半段用旧权重、后半段用新权重。

**不变量与失败模式：** `update_weights_interval` 不能为 0；被 `ray.get` 收束的 next future 已经生成的数据会留到后续循环使用，不能丢失语义。

**Comment：** 默认每步更新权重时，async 会频繁收束预取；更大的 interval 才能释放更多重叠空间。

## 4. periodic action：save / eval 的触发规则

### 4.1 should_run_periodic_action

**问题与约束：** save、eval 等动作既可以按固定 interval 触发，也应该在 epoch 边界和最后一轮触发；`rollout_id` 又是 0-based。

**设计选择：** `interval is None` 直接关闭；如果提供 `num_rollout` 且当前是最后一轮，强制触发；否则用 `step = rollout_id + 1` 做 interval 和 epoch boundary 判断。

**Explain：** 这个 helper 把周期动作的边界语义统一起来，避免每个调用点自己处理最后一轮。

来源：slime/utils/misc.py L119-L126

**Code：**

```python
if interval is None:
    return False

if num_rollout is not None and rollout_id == num_rollout - 1:
    return True

step = rollout_id + 1
return (step % interval == 0) or (num_rollout_per_epoch is not None and step % num_rollout_per_epoch == 0)
```

**代码逻辑：** 空 interval 表示不运行；最后一轮优先返回 True；常规路径把 0-based rollout 转成 1-based step 后做整除判断。

**为什么这样写：** 最后一轮 checkpoint / eval 是训练收尾安全网；使用 1-based step 可以让“每 N 轮”符合用户直觉。

**不变量与失败模式：** 调用方如果不传 `num_rollout`，最后一轮强制触发不会生效；`interval` 必须是正整数或 None。

**Comment：** 同步和异步训练都复用这个 helper，因此周期语义是一致的。

## 5. 走读小结

同步版 `train.py` 强调严格时序：generate 完成后 train，train 后保存和权重同步，再 eval。它能处理 colocate/offload，因为每个显存交接点都被串行化。

异步版 `train_async.py` 强调预取：当前 step 训练时提前启动下一轮 generate，但因此禁止 colocate，并且权重更新前必须先等待正在预取的 generate 完成。

两条路径共享的核心不变量是：训练数据、`rollout_id`、权重版本和 save/eval 周期必须对齐。
