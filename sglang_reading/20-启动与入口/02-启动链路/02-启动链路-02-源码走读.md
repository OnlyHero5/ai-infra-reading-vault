# 启动链路与 CLI · 源码走读

> 读法：`sglang serve` 的启动链路先由 CLI 根命令保留子命令未知参数，再由 `serve()` 做 help、插件加载、模型类型分发，LLM 路径进入 `ServerArgs` 解析与 `run_server`。旧式 `python -m sglang.launch_server` 仍存在，但推荐入口已经是 `sglang serve`。

---

## 1. CLI 根路由与 serve 分发

### 1.1 `main()`：子命令路由与未知参数透传

来源：python/sglang/cli/main.py L1-L46

**问题与约束：** `sglang` 根命令需要支持多个子命令，但 `serve` 的参数面非常大，根 parser 不能提前解析全部服务参数。

**设计选择：** 根 parser 只注册 `serve`、`generate`、`version` 三个子命令，并用 `parse_known_args()` 把未知参数保留为 `extra_argv` 交给子模块。

**Explain：** 这里的关键是根命令只判断“走哪条入口”，不解释 `--model-path`、`--tp-size` 等服务参数。这样 `serve` 可以独立维护自己的 parser。

**Code：**

```python
import argparse

from sglang.cli.utils import get_git_commit_hash
from sglang.version import __version__


def version(args, extra_argv):
    print(f"sglang version: {__version__}")
    print(f"git revision: {get_git_commit_hash()[:7]}")


def main():
    parser = argparse.ArgumentParser()

    # complex sub commands
    subparsers = parser.add_subparsers(dest="subcommand", required=True)
    subparsers.add_parser(
        "serve",
        help="Launch an SGLang server.",
        add_help=False,
    )
    subparsers.add_parser(
        "generate",
        help="Run inference on a multimodal model.",
        add_help=False,
    )

    # simple commands
    version_parser = subparsers.add_parser(
        "version",
        help="Show the version information.",
    )
    version_parser.set_defaults(func=version)

    args, extra_argv = parser.parse_known_args()

    if args.subcommand == "serve":
        from sglang.cli.serve import serve

        serve(args, extra_argv)
    elif args.subcommand == "generate":
        from sglang.cli.generate import generate

        generate(args, extra_argv)
    elif args.subcommand == "version":
        version(args, extra_argv)
```

**代码逻辑：** 函数创建根 parser 和 subparsers，要求必须有子命令；`serve`/`generate` 关闭子命令层 help，由后续模块处理。解析后根据 `args.subcommand` 延迟 import 并调用对应函数。

**为什么这样写：** 延迟 import 可以避免 `version` 这类轻量命令加载服务端重依赖；未知参数透传让 `serve` parser 可以单独演进，不被根 parser 限制。

**不变量与失败模式：** 根命令必须带子命令；`extra_argv` 必须原样保留服务参数；`version` 不应进入服务启动链路。若根 parser 直接解析 serve 参数，diffusion/LLM 两套参数面会互相污染。

**Comment：** `main()` 的边界很清楚：只做子命令路由，不做服务配置解析。

### 1.2 `_extract_model_type_override`：剥离 CLI 分发 hint

来源：python/sglang/cli/serve.py L16-L46

**问题与约束：** `--model-type` 用于 CLI 层选择 LLM 或 diffusion 路径，但它不是 `ServerArgs` 字段，不能传给 LLM 的 argparse。

**设计选择：** 手动扫描 `extra_argv`，支持 `--model-type value` 和 `--model-type=value` 两种写法；把该参数从 argv 中移除，并校验值只能是 `auto/llm/diffusion`。

**Explain：** 这是 serve 层自己的分发参数。过滤后的 `dispatch_argv` 才能安全传给 LLM 或 diffusion 的各自 parser。

**Code：**

```python
def _extract_model_type_override(extra_argv):
    """Extract and remove --model-type override from argv."""
    model_type = "auto"
    filtered_argv = []
    i = 0
    while i < len(extra_argv):
        arg = extra_argv[i]
        if arg == "--model-type":
            if i + 1 >= len(extra_argv):
                raise Exception(
                    "Error: --model-type requires a value. "
                    "Valid values are: auto, llm, diffusion."
                )
            model_type = extra_argv[i + 1]
            i += 2
            continue

        if arg.startswith("--model-type="):
            model_type = arg.split("=", 1)[1]
            i += 1
            continue

        filtered_argv.append(arg)
        i += 1

    if model_type not in ("auto", "llm", "diffusion"):
        raise Exception(
            f"Error: invalid --model-type '{model_type}'. "
            "Valid values are: auto, llm, diffusion."
        )
    return model_type, filtered_argv
```

**代码逻辑：** 循环扫描 argv，遇到两种 model-type 形式就提取值并跳过原参数；其他参数进入 `filtered_argv`。最后校验枚举并返回。

**为什么这样写：** `--model-type` 是 serve dispatcher 的私有参数，不属于 LLM server args。先剥离可以避免 `prepare_server_args` 遇到未知参数报错。

**不变量与失败模式：** 缺失值和非法值都必须报错；未匹配参数顺序要保持；默认是 `auto`。若不剥离，LLM 路径会把 `--model-type` 当未知参数。

**Comment：** 分发 hint 和 server 配置不是同一层参数，这个函数负责把两者拆开。

### 1.3 `serve()`：help、插件、类型检测与最终分支

来源：python/sglang/cli/serve.py L49-L130

**问题与约束：** `sglang serve` 可能启动 LLM server 或 diffusion server；help 需要展示两套参数；正常启动要先加载插件，再决定进入哪条路径，并在退出时清理子进程。

**设计选择：** help 分支打印总说明并分别调用 LLM/diffusion parser；正常分支先 `load_plugins()`，再提取 model type、读取 model path、自动检测或使用 override，最后分发到 diffusion `execute_serve_cmd` 或 LLM `prepare_server_args + run_server`。

**Explain：** `serve()` 是 CLI 到 runtime 的编排函数。它是唯一同时知道插件阶段、模型类型判断和两套 server parser 的入口。

**Code：**

```python
def serve(args, extra_argv):
    if any(h in extra_argv for h in ("-h", "--help")):
        print(
            "Usage: sglang serve --model-path <model-name-or-path> [additional-arguments]\n\n"
            "This command can launch either a standard language model server or a diffusion model server.\n"
            "The server type is determined by the --model-path.\n"
            "Optional override: --model-type {auto,llm,diffusion} "
            "(default: auto, fallback to LLM on detection failure)."
        )

        print("\n--- Help for Standard Language Model Server ---")
        from sglang.srt.server_args import prepare_server_args

        try:
            prepare_server_args(["--help"])
        except SystemExit:
            pass  # argparse --help calls sys.exit

        print("\n--- Help for Diffusion Model Server ---")
        try:
            from sglang.multimodal_gen.runtime.entrypoints.cli.serve import (
                add_multimodal_gen_serve_args,
            )

            parser = argparse.ArgumentParser(
                prog="sglang serve",
                description="SGLang Diffusion Model Serving",
            )
            add_multimodal_gen_serve_args(parser)
            parser.print_help()
        except ImportError:
            print(
                "Diffusion model support is not available. "
                'Install with: pip install "sglang[diffusion]"'
            )
        return

    from sglang.srt.plugins import load_plugins

    load_plugins()

    model_type, dispatch_argv = _extract_model_type_override(extra_argv)
    model_path = get_model_path(dispatch_argv)
    try:
        if model_type == "auto":
            is_diffusion_model = get_is_diffusion_model(model_path)
            if is_diffusion_model:
                logger.info("Diffusion model detected")
        else:
            is_diffusion_model = model_type == "diffusion"
            logger.info(
                "Dispatch override enabled: --model-type=%s " "(skip auto detection)",
                model_type,
            )

        if is_diffusion_model:
            from sglang.multimodal_gen.runtime.entrypoints.cli.serve import (
                add_multimodal_gen_serve_args,
                execute_serve_cmd,
            )

            parser = argparse.ArgumentParser(
                description="SGLang Diffusion Model Serving"
            )
            add_multimodal_gen_serve_args(parser)
            parsed_args, remaining_argv = parser.parse_known_args(dispatch_argv)

            execute_serve_cmd(parsed_args, remaining_argv)
        else:
            from sglang.launch_server import run_server
            from sglang.srt.server_args import prepare_server_args

            server_args = prepare_server_args(dispatch_argv)

            run_server(server_args)
    finally:
        kill_process_tree(os.getpid(), include_parent=False)
```

**代码逻辑：** help 路径提前返回；正常路径先加载插件，再得到 `model_type` 与过滤后的 argv。自动模式调用检测函数，override 模式直接设定布尔值。diffusion 分支使用 diffusion parser，LLM 分支使用 `ServerArgs` 工厂，finally 统一清理子进程。

**为什么这样写：** LLM 与 diffusion 的参数集合不同，不能用一个 parser 强行覆盖。插件必须在服务 runtime import 和参数构造前加载，才能注册 hook 或替换类。

**不变量与失败模式：** help 不应要求真实 model path；插件加载要早于 `prepare_server_args`；finally 必须覆盖两条启动路径。若检测失败，默认 fallback 到 LLM 路径，由后续模型加载暴露更具体错误。

**Comment：** `serve()` 把启动链路分成三段：展示帮助、准备扩展点、进入具体 runtime。

### 1.4 `get_model_path`：在 server parser 之前提取模型路径

来源：python/sglang/cli/utils.py L102-L127

**问题与约束：** 模型类型自动检测需要 `--model-path`，但此时还没进入 `prepare_server_args`；CLI 还支持 `--model` 别名和 `--flag=value` 形式。

**设计选择：** 手动扫描 argv，支持 `--model-path`、`--model` 及等号形式；缺少路径时按 help/非 help 场景抛不同错误。

**Explain：** 这是 serve dispatcher 的早期参数读取，只为模型类型检测服务。

**Code：**

```python
def get_model_path(extra_argv):
    # Find the model_path argument
    model_path = None
    for i, arg in enumerate(extra_argv):
        if arg in ("--model-path", "--model"):
            if i + 1 < len(extra_argv):
                model_path = extra_argv[i + 1]
                break
        elif arg.startswith("--model-path=") or arg.startswith("--model="):
            model_path = arg.split("=", 1)[1]
            break

    if model_path is None:
        # Fallback for --help or other cases where model-path is not provided
        if any(h in extra_argv for h in ["-h", "--help"]):
            raise Exception(
                "Usage: sglang serve --model-path <model-name-or-path> [additional-arguments]\n\n"
                "This command can launch either a standard language model server or a diffusion model server.\n"
                "The server type is determined by the --model-path.\n"
            )
        else:
            raise Exception(
                "Error: --model-path is required. "
                "Please provide the path to the model."
            )
    return model_path
```

**代码逻辑：** 函数遍历 argv 找两类模型路径参数；找到后返回字符串。未找到时，help 场景给 usage，普通场景给必填错误。

**为什么这样写：** 只有拿到 model path 才能判断 diffusion/LLM，而完整 argparse 还要等路径分发之后才知道使用哪套 parser。

**不变量与失败模式：** 过滤后的 argv 中必须仍保留 model path；别名和等号形式都要支持；非 help 启动缺 model path 必须失败。若默认空路径继续走检测，会把配置错误伪装成模型类型判断失败。

**Comment：** 这是一个早期窥探参数的 helper，不替代 `ServerArgs` 的完整解析。

### 1.5 `get_is_diffusion_model`：自动检测并 fail-open 到 LLM

来源：python/sglang/cli/utils.py L59-L99

**问题与约束：** 用户传入的 model path 可能是本地目录、HF/ModelScope repo，或注册表中的非 diffusers diffusion 模型；网络、权限或离线环境下检测可能失败。

**设计选择：** 先查 overlay、本地目录、known non-diffusers 和 registry；远程路径只尝试下载 `model_index.json`；任何异常都记录 debug 并返回 False。

**Explain：** False 并不表示模型一定是 LLM，只表示自动检测没有确认 diffusion。后续 LLM 加载仍可能失败并给出更具体错误。

**Code：**

```python
def get_is_diffusion_model(model_path: str) -> bool:
    """Detect whether model_path points to a diffusion model.

    For local directories, checks the filesystem directly.
    For HF/ModelScope model IDs, attempts to fetch only model_index.json.
    For gated repos where file download fails, falls back to HF model card
    metadata (library_name == "diffusers").
    Returns False on any failure (network error, 404, offline mode, etc.)
    so that the caller falls through to the standard LLM server path.
    """
    if _is_overlay_diffusion_model(model_path):
        # short-circuit, if applicable for the overlay mechanism (diffusion-only)
        return True

    if os.path.isdir(model_path):
        if _is_diffusers_model_dir(model_path):
            return True
        return is_known_non_diffusers_diffusion_model(model_path)

    if is_known_non_diffusers_diffusion_model(model_path):
        return True

    if _is_registered_diffusion_model(model_path):
        return True

    try:
        if envs.SGLANG_USE_MODELSCOPE.get():
            from modelscope import model_file_download

            file_path = model_file_download(
                model_id=model_path, file_path="model_index.json"
            )
        else:
            from huggingface_hub import hf_hub_download

            file_path = hf_hub_download(repo_id=model_path, filename="model_index.json")

        return _is_diffusers_model_dir(os.path.dirname(file_path))
    except Exception as e:
        logger.debug("Failed to auto-detect diffusion model for %s: %s", model_path, e)
        return False
```

**代码逻辑：** 函数按本地/注册/远程顺序判断。远程源根据环境变量选择 ModelScope 或 HF Hub，只下载 model index。异常时不抛给调用方。

**为什么这样写：** 自动检测是便利功能，不能因为网络不可达阻断本可走 LLM 的启动。用户仍可用 `--model-type=diffusion` 强制走 diffusion 路径。

**不变量与失败模式：** 本地目录判断优先；远程失败必须 fail-open；返回 True 才进入 diffusion parser。若检测失败时直接抛错，普通 LLM repo 在离线环境也会被误阻断。

**Comment：** diffusion 检测是保守确认，不是强制分类器。

## 2. 插件与 Hook 阶段

### 2.1 `load_plugins_by_group`：entry point 发现、白名单与平台过滤

来源：python/sglang/srt/plugins/__init__.py L35-L86

**问题与约束：** 已安装包可以通过 entry point 注册插件，但某些平台插件依赖硬件库，不能在未选中平台时导入；用户还可能只想加载白名单插件。

**设计选择：** 用 `entry_points(group=...)` 发现插件；`SGLANG_PLUGINS` 作为名字白名单；`excluded_dists` 跳过未选中平台包；单个插件 load 失败只记录异常。

**Explain：** 这是插件发现层，只负责返回插件 callable 和分发包名，不执行插件逻辑。

**Code：**

```python
def load_plugins_by_group(
    group: str,
    excluded_dists: set[str] | None = None,
) -> dict[str, tuple[Callable[[], Any], str | None]]:
    """
    Discover and load plugins registered under the given entry point group.
    """
    # SGLANG_PLUGINS whitelist (comma-separated plugin names)
    allowed_set: set[str] | None = None
    allowed_str = envs.SGLANG_PLUGINS.get()
    if allowed_str:
        allowed_set = {x.strip() for x in allowed_str.split(",") if x.strip()}

    discovered = entry_points(group=group)
    if len(discovered) == 0:
        logger.debug("No plugins found for group %s.", group)
        return {}

    logger.info("Available plugins for group %s:", group)
    for ep in discovered:
        logger.info("  - %s -> %s", ep.name, ep.value)

    plugins: dict[str, tuple[Callable[[], Any], str | None]] = {}
    for ep in discovered:
        if allowed_set is not None and ep.name not in allowed_set:
            logger.info("Skipping plugin %s (not in SGLANG_PLUGINS)", ep.name)
            continue
        dist_name = ep.dist.name if ep.dist else None
        if excluded_dists and dist_name in excluded_dists:
            logger.info(
                "Skipping plugin %s (dist %s excluded by SGLANG_PLATFORM)",
                ep.name,
                dist_name,
            )
            continue
        try:
            func = ep.load()
            plugins[ep.name] = (func, dist_name)
            logger.info("Loaded plugin %s from group %s", ep.name, group)
        except Exception:
            logger.exception("Failed to load plugin %s from group %s", ep.name, group)

    return plugins
```

**代码逻辑：** 函数构造可选白名单，读取 entry points，逐个按白名单和平台排除集过滤，成功 load 后放入 dict。

**为什么这样写：** 插件发现本身可能导入第三方依赖。平台过滤能避免未使用的硬件插件拉起不必要依赖；单插件失败不应拖垮整个服务。

**不变量与失败模式：** `ep.load()` 只对通过过滤的插件执行；返回 dict key 是插件名；异常必须被捕获并继续。若未过滤平台包，CPU/CUDA 环境可能因为无关硬件库 import 失败。

**Comment：** 插件系统先把“能加载哪些 callable”算出来，是否执行由上层 `load_plugins()` 决定。

### 2.2 `load_plugins`：执行插件并一次性 apply hooks

来源：python/sglang/srt/plugins/__init__.py L103-L141

**问题与约束：** 调用方希望一个函数完成通用插件执行和 Hook 应用；插件函数的副作用是注册 hook、替换类等，返回值不重要；同一进程可能多次调用加载函数。

**设计选择：** 用 `_plugins_loaded` 保证幂等；加载 general plugin group 后逐个执行，并用 `_current_plugin_source` 记录插件来源；最后统一 `HookRegistry.apply_hooks()`。

**Explain：** `serve()` 只调用 `load_plugins()`，不直接碰 hook registry。这个函数把插件执行和 hook monkey-patch 时机固定在引擎启动前。

**Code：**

```python
def load_plugins():
    """
    Load and execute all general plugins, then apply registered hooks.

    Idempotent - safe to call multiple times. General plugins are functions
    whose side effects (registering hooks, replacing classes, etc.) are the
    desired behavior. Return values are ignored.
    """
    global _plugins_loaded
    if _plugins_loaded:
        return
    _plugins_loaded = True

    plugins = load_plugins_by_group(
        GENERAL_PLUGINS_GROUP,
        excluded_dists=_get_excluded_dists(),
    )

    for name, (func, dist_name) in plugins.items():
        source = HookSource(plugin_name=name, dist_name=dist_name)
        token = _current_plugin_source.set(source)
        try:
            func()
            logger.info("Executed general plugin: %s", name)
        except Exception:
            logger.exception("Failed to execute general plugin: %s", name)
        finally:
            _current_plugin_source.reset(token)

    # Apply all registered hooks (idempotent — already-patched targets are skipped).
    HookRegistry.apply_hooks()
```

**代码逻辑：** 函数先做幂等检查；加载插件 callable；执行每个插件前设置 context var，结束后 reset；所有插件执行完成后 apply hooks。

**为什么这样写：** Hook 注册要先收集完，再统一 patch，否则插件之间的替换顺序和 stale binding 更难控制。context var 让注册日志能追踪来自哪个插件。

**不变量与失败模式：** `_plugins_loaded` 必须阻止重复执行插件；插件异常不阻止后续插件；hooks 应在插件全部执行后应用。若每个插件执行后立即 patch，后注册的 REPLACE/AROUND 顺序会更难预测。

**Comment：** `load_plugins()` 是启动链路中的扩展点收束：执行插件副作用，然后应用 monkey patch。

### 2.3 `HookRegistry.register`：Hook 类型和注册校验

来源：python/sglang/srt/plugins/hook_registry.py L60-L143

**问题与约束：** 插件需要对函数、方法或类做 before/after/around/replace 扩展；类替换和函数 hook 的语义不同，重复替换也需要可见日志。

**设计选择：** 定义四种 `HookType`；`register` 接收 target dotted path、hook callable 和类型；类对象只能用于 REPLACE；重复 REPLACE 记录 warning；注册时附带插件来源。

**Explain：** 这是插件作者面对的核心 API。target 是完全限定路径，后续 apply 阶段再解析并 patch。

**Code：**

```python
class HookType(Enum):
    """Types of hooks that can be applied to functions or classes."""

    BEFORE = "before"  # Execute before original; can modify args
    AFTER = "after"  # Execute after original; can modify return value
    AROUND = "around"  # Wrap original; full control over execution
    REPLACE = "replace"  # Replace the original function or class entirely


class HookRegistry:
    """
    Global registry for function/method/class hooks.
    """

    _hooks: dict[str, list[tuple[HookType, Callable, HookSource | None]]] = defaultdict(
        list
    )
    _patched: set[str] = set()

    @classmethod
    def register(
        cls,
        target: str,
        hook: Callable,
        hook_type: HookType = HookType.AFTER,
        *,
        source: HookSource | None = None,
    ):
        """
        Register a hook on a target function, method, or class.
        """
        if isinstance(hook, type) and hook_type != HookType.REPLACE:
            raise TypeError(
                f"Class {hook.__name__} can only be used with HookType.REPLACE, "
                f"got HookType.{hook_type.name}. "
                f"Use a function for BEFORE/AFTER/AROUND hooks."
            )
        resolved_source = source or _current_plugin_source.get()
        # Warn on duplicate REPLACE for the same target
        if hook_type == HookType.REPLACE:
            existing_replace = [
                (h, src) for ht, h, src in cls._hooks[target] if ht == HookType.REPLACE
            ]
            if existing_replace:
                prev, prev_src = existing_replace[-1]
                prev_name = getattr(prev, "__qualname__", None) or repr(prev)
                new_name = getattr(hook, "__qualname__", None) or repr(hook)
                logger.warning(
                    "Multiple REPLACE hooks on '%s': previous (%s [%s]) will be "
                    "overridden by (%s [%s]). The last registered REPLACE takes effect.",
                    target,
                    prev_name,
                    _format_source(prev_src),
                    new_name,
                    _format_source(resolved_source),
                )
        cls._hooks[target].append((hook_type, hook, resolved_source))
        logger.debug(
            "Registered %s hook on %s [%s]",
            hook_type.value,
            target,
            _format_source(resolved_source),
        )
```

**代码逻辑：** 枚举定义四类 hook。注册函数先检查类对象只能替换；然后解析来源，检查重复 REPLACE，最后把 `(hook_type, hook, source)` 放入目标列表。

**为什么这样写：** 类替换会改变对象身份，不能当 BEFORE/AFTER 函数使用。重复 REPLACE 最终只有最后一个有效，因此必须打 warning。

**不变量与失败模式：** target 必须是后续可解析的 dotted path；类只能 REPLACE；注册只记录，不立即 patch。若注册阶段直接 patch，插件执行顺序会影响后续插件能否看到原始对象。

**Comment：** HookRegistry 先收集意图，真正修改对象放到 `apply_hooks()`。

### 2.4 `apply_hooks`：排序后一次性 patch target

来源：python/sglang/srt/plugins/hook_registry.py L145-L166

**问题与约束：** 注册的 hook 需要在服务启动前应用；类 REPLACE 要先于子方法 hook，避免方法 hook 绑定到旧类；同一 target 不能重复 patch。

**设计选择：** 按 `_target_sort_key` 排序 `_hooks`，跳过已 patch target，调用 `_apply_target`，成功后加入 `_patched`。

**Explain：** 这是 hook 系统从“注册表”变成“实际 monkey patch”的时刻。

**Code：**

```python
    @classmethod
    def apply_hooks(cls):
        """
        Apply all registered hooks to their target functions/classes.

        This performs the actual monkey-patching. Should be called once after
        all plugins have been loaded and before the engine starts.

        Targets with class REPLACE hooks are applied first, so that
        subsequent method-level hooks (AROUND, BEFORE, AFTER) on child
        attributes resolve against the *replaced* class rather than the
        original.
        """
        sorted_items = sorted(cls._hooks.items(), key=cls._target_sort_key)
        for target, hooks in sorted_items:
            if target in cls._patched:
                continue
            try:
                cls._apply_target(target, hooks)
                cls._patched.add(target)
            except Exception:
                logger.exception("Failed to apply hooks to %s", target)
```

**代码逻辑：** 函数排序所有 hook target；对每个 target，如果已经 patch 就跳过，否则尝试应用并记录已 patch。

**为什么这样写：** 类替换和方法 hook 的先后关系会影响最终对象。统一排序加幂等集合能让插件阶段可重复调用但不重复修改对象。

**不变量与失败模式：** `apply_hooks` 应在 engine serving 前调用；失败 target 要记录异常；已 patch target 不重复执行。若方法 hook 先于类 REPLACE，替换后的类可能丢掉方法级 hook。

**Comment：** apply 阶段是启动时一次性动作，不是请求时动态注册机制。

### 2.5 `_wrap_fn`：四种 Hook 的运行时包装语义

来源：python/sglang/srt/plugins/hook_registry.py L352-L393

**问题与约束：** BEFORE/AFTER/AROUND/REPLACE 四种 hook 要映射成实际可调用 wrapper，并保留原函数引用供调试或二次包装识别。

**设计选择：** REPLACE wrapper 直接调用 hook；BEFORE 允许 hook 返回新 `(args, kwargs)`；AFTER 允许 hook 修改 result；AROUND 把 original_fn 交给 hook；每个 wrapper 设置 `__wrapped__`。

**Explain：** 这段定义了 HookType 的真正执行语义，而不仅是枚举名称。

**Code：**

```python
def _wrap_fn(original_fn: Callable, hook: Callable, hook_type: HookType) -> Callable:
    """Create a wrapper function based on the hook type."""
    if hook_type == HookType.REPLACE:

        @functools.wraps(original_fn)
        def wrapper(*args, **kwargs):
            return hook(*args, **kwargs)

        wrapper.__wrapped__ = original_fn
        return wrapper

    elif hook_type == HookType.BEFORE:

        @functools.wraps(original_fn)
        def wrapper(*args, **kwargs):
            result = hook(*args, **kwargs)
            if result is not None:
                args, kwargs = result
            return original_fn(*args, **kwargs)

        wrapper.__wrapped__ = original_fn
        return wrapper

    elif hook_type == HookType.AFTER:

        @functools.wraps(original_fn)
        def wrapper(*args, **kwargs):
            result = original_fn(*args, **kwargs)
            modified = hook(result, *args, **kwargs)
            return modified if modified is not None else result

        wrapper.__wrapped__ = original_fn
        return wrapper

    elif hook_type == HookType.AROUND:

        @functools.wraps(original_fn)
        def wrapper(*args, **kwargs):
            return hook(original_fn, *args, **kwargs)

        wrapper.__wrapped__ = original_fn
        return wrapper
```

**代码逻辑：** 函数根据 hook type 返回不同 wrapper；每类 wrapper 都闭包捕获 original_fn 和 hook，并把原函数挂到 `__wrapped__`。

**为什么这样写：** 不同 hook 类型需要不同控制权：BEFORE 修改输入，AFTER 修改输出，AROUND 全权控制，REPLACE 完全替换。统一包装函数可以让 `_apply_target` 只关心给目标赋新 callable。

**不变量与失败模式：** BEFORE hook 返回值若非 None 必须是 `(args, kwargs)`；AFTER 返回 None 表示保留原结果；AROUND hook 自己负责是否调用 original。若 wrapper 不保留 `__wrapped__`，调试和防重复包装会更难。

**Comment：** Hook 系统的扩展能力最终落在这些 wrapper 的调用约定上。

### 3.2 `prepare_server_args`：argv 到 ServerArgs 的工厂函数

来源：python/sglang/srt/server_args.py L7561-L7595

**问题与约束：** LLM 路径需要从 argv 构造 `ServerArgs`；还要支持 YAML config 合并，并且 `ServerArgs.__post_init__` 中的 log 需要在构造前就有格式化配置。

**设计选择：** 创建 `argparse.ArgumentParser(prog="sglang serve")`，注册 CLI 参数；如果 argv 含 `--config`，用 `ConfigArgumentMerger` 合并；parse 后先初始化 logging，再调用 `ServerArgs.from_cli_args`。

**Explain：** 这是 LLM 路径最关键的参数工厂，serve 层只负责把过滤后的 argv 传进来。

**Code：**

```python
def prepare_server_args(argv: List[str]) -> ServerArgs:
    """
    Prepare the server arguments from the command line arguments.
    """
    parser = argparse.ArgumentParser(prog="sglang serve")
    ServerArgs.add_cli_args(parser)

    # Check for config file and merge arguments if present
    if "--config" in argv:
        # Import here to avoid circular imports
        from sglang.srt.server_args_config_parser import ConfigArgumentMerger

        # Extract boolean actions from the parser to handle them correctly
        config_merger = ConfigArgumentMerger(parser)
        argv = config_merger.merge_config_with_args(argv)

    raw_args = parser.parse_args(argv)

    # Set up basic logging before ServerArgs.__post_init__ so that
    # logger.info / logger.warning calls there are properly formatted.
    logging.basicConfig(
        level=getattr(logging, raw_args.log_level.upper()),
        format="[%(asctime)s] %(message)s",
        datefmt="%Y-%m-%d %H:%M:%S",
        force=True,
    )

    return ServerArgs.from_cli_args(raw_args)
```

**代码逻辑：** 函数创建 parser、注册参数、可选合并 config，再 parse argv。parse 之后用 `raw_args.log_level` 初始化 logging，最后把 Namespace 转 dataclass。

**为什么这样写：** YAML config 合并需要知道 parser 的 action 类型，尤其是 bool flag；logging 要早于 post-init，否则 post-init 的 warning/info 格式不稳定。

**不变量与失败模式：** `--config` 合并要在 `parse_args` 前；CLI 应覆盖 config 中同名值；`log_level` 必须能映射到 logging level。若 logging 初始化放在 `from_cli_args` 后，post-init 期间日志可能缺格式。

**Comment：** `prepare_server_args` 是 CLI 参数进入 SRT runtime 的正式边界。

### 3.3 `from_cli_args`：Namespace 到 dataclass

来源：python/sglang/srt/server_args.py L6862-L6869

**问题与约束：** 并不是所有 `ServerArgs` dataclass 字段都有 CLI 参数；直接从 Namespace 取所有字段会访问不存在属性。

**设计选择：** 遍历 dataclass fields，只保留 Namespace 上实际存在的字段，然后用这些属性构造 `ServerArgs`。

**Explain：** 无 CLI 面的字段保留 dataclass 默认值，避免 argparse 需要知道内部字段。

**Code：**

```python
    def from_cli_args(cls, args: argparse.Namespace):
        # Some dataclass fields (e.g. stat_loggers) intentionally have no CLI
        # surface and won't appear on the argparse Namespace. Skip them so the
        # dataclass default applies.
        attrs = [
            attr.name for attr in dataclasses.fields(cls) if hasattr(args, attr.name)
        ]
        return cls(**{attr: getattr(args, attr) for attr in attrs})
```

**代码逻辑：** 函数按 dataclass 字段名过滤 Namespace 属性，构建 dict 后调用 dataclass constructor。

**为什么这样写：** `ServerArgs` 同时包含 CLI 配置和运行时内部字段。过滤缺失属性可以保持内部字段不暴露到 CLI。

**不变量与失败模式：** 构造 dataclass 时会触发 `__post_init__`；缺失 CLI 字段必须有默认值；字段名和 argparse dest 要一致。若无过滤直接 `getattr`，内部字段会导致解析阶段崩溃。

**Comment：** 这一步完成数据结构转换，同时保留 dataclass 默认值机制。

### 3.4 `__post_init__`：第二遍校验与默认值编排

来源：python/sglang/srt/server_args.py L2567-L2616

**问题与约束：** CLI parse 只能得到原始字段值；真正启动前还要下载/规范化模型、校验 SSL/PD/ASR/CP/DCP、处理 dummy model 和 deprecated 参数。

**设计选择：** `__post_init__` 作为构造后的统一处理入口，先执行与模型无关的校验，再处理 dummy model early return，随后处理 deprecated 参数和 `quantization == "unquant"` 等语义转换。

**Explain：** `ServerArgs` 不是纯数据容器，构造后会做第二遍语义加工。后续服务启动拿到的是已经处理过的配置。

**Code：**

```python
    def __post_init__(self):
        """
        Orchestrates the handling of various server arguments, ensuring proper configuration and validation.
        """

        self._maybe_download_model_for_runai()

        self._handle_load_balance_method()

        # Validate mm_process_config before dummy-model early return.
        self._handle_multimodal()
        # Validate SSL arguments early (before dummy-model short-circuit).
        self._handle_ssl_validation()
        # Validate transcription/ASR-specific server args (model-independent).
        self._handle_asr_validation()

            handle_pd_disaggregation,
        )

        if self.enable_session_radix_cache and self.radix_eviction_policy != "priority":
            raise ValueError(
                "--enable-session-radix-cache requires --radix-eviction-policy priority"
            )

        # Normalize deprecated CP aliases before validations or model-specific
        # defaults inspect enable_prefill_cp/cp_strategy.
        self._handle_legacy_cp_arguments()
        self._validate_prefill_only_disable_kv_cache_args()
        self._handle_dcp_validation()

        if self.model_path.lower() in ["none", "dummy"]:
            # Skip for dummy models
            return

        # Handle deprecated arguments.
        self._handle_deprecated_args()

        # Handle deprecated environment variables for prefill delayer.
        self._handle_prefill_delayer_env_compat()

        # Convert to None now (before model config validation), but record
        # the intent so auto-detection in _handle_model_specific_adjustments

        if self.quantization == "unquant":

```

**代码逻辑：** 方法按固定顺序调用多个 handler；session radix cache 与 eviction policy 有显式 ValueError；dummy model 在部分校验后提前返回；`unquant` 被转成 `None`。

**为什么这样写：** 启动配置的约束跨多个字段，难以在 argparse 层表达。post-init 能在 dataclass 已有完整字段后做跨字段校验和默认值推导。

**不变量与失败模式：** 模型无关校验要在 dummy return 前执行；`unquant` 的用户意图要在后续模型自动检测前记录；不兼容组合必须直接报错。若字段语义只停留在字符串值，后续 runtime 分支会反复处理同一逻辑。

**Comment：** `__post_init__` 是 ServerArgs 的语义层，parse 只是它的输入。

## 4. 进入 runtime

### 4.1 `run_server`：按 server args 选择具体服务入口

来源：python/sglang/launch_server.py L15-L51

**问题与约束：** LLM server 不是只有默认 HTTP 一条路径；encoder-only、legacy gRPC、Ray HTTP 和默认 HTTP 都需要从同一个 `ServerArgs` 入口分派。

**设计选择：** `run_server` 先检查 `encoder_only`，再检查 `grpc_mode`，再检查 `use_ray`，最后落到默认 HTTP server。

**Explain：** 这是 `prepare_server_args` 之后的 runtime 分发点。CLI 到这里已经完成参数解析，剩下按配置选择启动模块。

**Code：**

```python
def run_server(server_args):
    """Run the server based on server_args.grpc_mode and server_args.encoder_only."""
    if server_args.encoder_only:
        # For encoder disaggregation
        if server_args.grpc_mode:
            from sglang.srt.disaggregation.encode_grpc_server import (
                serve_grpc_encoder,
            )

            asyncio.run(serve_grpc_encoder(server_args))
        else:
            from sglang.srt.disaggregation.encode_server import launch_server

            launch_server(server_args)
    elif server_args.grpc_mode:
        # TODO: Once the native Rust gRPC server starts alongside HTTP in the
        # default path below (controlled by SGLANG_ENABLE_GRPC / SGLANG_GRPC_PORT),
        # remove this legacy SMG path and the grpc_mode flag.
        from sglang.srt.entrypoints.grpc_server import serve_grpc

        asyncio.run(serve_grpc(server_args))
    elif server_args.use_ray:
        # Ray mode: HTTP mode with Ray backend.
        try:
            from sglang.srt.ray.http_server import launch_server
        except ImportError:
            raise ImportError(
                "Ray is required for --use-ray mode. "
                "Install it with: pip install 'sglang[ray]'"
            )

        launch_server(server_args)
    else:
        # Default mode: HTTP mode.
        from sglang.srt.entrypoints.http_server import launch_server

        launch_server(server_args)
```

**代码逻辑：** 函数按条件链选择模块并延迟 import。encoder+grpc 用 encoder gRPC；encoder 非 gRPC 用 encoder server；普通 gRPC 用 grpc server；Ray 模式导入 Ray HTTP server；默认导入普通 HTTP server。

**为什么这样写：** 不同启动模式依赖不同模块和可选依赖。延迟 import 让默认路径不加载 Ray 或 gRPC 相关依赖，也让缺 Ray 的错误只在 `--use-ray` 时出现。

**不变量与失败模式：** 条件顺序很重要，encoder-only 优先于普通 grpc；Ray 缺依赖时要给明确安装提示；默认路径必须是 HTTP server。若 `grpc_mode` 先于 encoder-only，encoder gRPC 会被错误分到普通 gRPC。

**Comment：** `run_server` 是 LLM 路径最后的启动分叉，下一篇 HTTP Server 再展开默认分支内部。

### 4.2 `python -m sglang.launch_server`：旧入口兼容

来源：python/sglang/launch_server.py L54-L72

**问题与约束：** 旧式 `python -m sglang.launch_server` 仍需兼容，但官方推荐 `sglang serve`；旧入口也要加载插件、解析参数并清理子进程。

**设计选择：** `__main__` 分支先发 warning，再调用 `load_plugins()`、`prepare_server_args(sys.argv[1:])`、`run_server(server_args)`，finally 中清理进程树。

**Explain：** 旧入口绕过 `cli/serve.py` 的 diffusion 自动检测和双 help 展示，但 LLM 路径的插件、ServerArgs 和 run_server 链路保持一致。

**Code：**

```python
if __name__ == "__main__":
    warnings.warn(
```

**代码逻辑：** 模块作为脚本运行时发出弃用式推荐提示；加载插件，解析 argv，运行 server，并在退出时清理子进程。

**为什么这样写：** 保留旧入口可以减少用户迁移成本；同时 warning 引导新用户使用 `sglang serve`，因为新入口拥有模型类型检测和更完整 help。

**不变量与失败模式：** 旧入口必须仍能启动 LLM server；插件加载要早于 `prepare_server_args`；finally 必须执行进程清理。若旧入口不加载插件，会与 `sglang serve` 的 LLM 行为不一致。

**Comment：** 旧入口是兼容层，不是新的主启动链路。
