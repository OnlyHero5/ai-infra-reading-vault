# MoE · 源码走读

> 走读主线：router 产生 top-k expert，TopK 选择具体路由算法并完成 EPLB 后处理，`FusedMoE` 组织 dispatch/core/combine，dispatcher 与 EPLB 负责跨 rank 通信、hook 与 logical→physical expert 映射。

---

## 1. Router：gate 与 top-k

### 1.1 cudacore kernel 内完成 gate、softcap、bias 与 top1

问题与约束：
- MoE router 对每个 token 都要计算 `num_experts` 个 logits；若先写出完整 logits 再调用通用 top-k，会增加显存写读与 launch 成本。

设计选择：
- 让一个 Triton program 处理一个 token，在寄存器里完成 hidden×gate、softcap、correction bias、top1 与 top1 权重写回。

Explain：
`fused_moe_router_cudacore_kernel` 直接加载一行 hidden state 和全部 expert router weight，计算 logits 后根据 `moe_softcapping` 做 tanh 形态压缩，再按需加 `correction_bias`。top1 id 与 top1 softmax 权重立即写入 `topk_ids/topk_weights`，避免 materialize 全量 router logits。

来源：python/sglang/srt/layers/moe/router.py L13-L76

Code：

```python
@triton.jit
def fused_moe_router_cudacore_kernel(
    input_ptr,
    moe_router_weight_ptr,
    topk_weights_ptr,
    topk_ids_ptr,
    correction_bias_ptr,
    is_correction_bias: tl.constexpr,
    num_experts: tl.constexpr,
    topk: tl.constexpr,
    moe_softcapping: tl.constexpr,
    moe_renormalize: tl.constexpr,
    hidden_dim: tl.constexpr,
    BLOCK_SIZE: tl.constexpr,
):
    pid = tl.program_id(axis=0)
    offsets = tl.arange(0, BLOCK_SIZE)
    mask = offsets < hidden_dim
    w_router = tl.load(
        moe_router_weight_ptr + expert_offsets * hidden_dim + offsets[None, :],
        mask=router_mask,
        other=0.0,
    )
    x = tl.load(input_ptr + pid * hidden_dim + offsets, mask=mask, other=0.0)
    logits = tl.sum((w_router.to(tl.float32) * x[None, :].to(tl.float32)), axis=-1)
    if moe_softcapping == 0:
        logits_softcapped = logits
    else:
        logits_scaled = logits / moe_softcapping
        logits_softcapped = (tl.exp(2 * logits_scaled) - 1) / (
            tl.exp(2 * logits_scaled) + 1
        ) * moe_softcapping
    if is_correction_bias:
        bias = tl.load(correction_bias_ptr + tl.arange(0, num_experts))
        logits_softcapped = logits_softcapped + bias
    top1 = tl.argmax(logits_softcapped, axis=0)
    tl.store(topk_ids_ptr + pid * topk + 0, top1)
    top1_v = tl.max(logits_softcapped, axis=0)
    invsumexp = 1.0 / tl.sum(tl.exp(logits_softcapped - top1_v), axis=0)
    tl.store(topk_weights_ptr + pid * topk + 0, invsumexp)
```

代码逻辑：
- program id 对应 token id；`BLOCK_SIZE` 覆盖 hidden dim；`mask` 处理 padding。
- logits 在 fp32 中累计，softcap 与 bias 都发生在 top-k 之前。
- `invsumexp` 用 top1 作为数值稳定基准，写出的 top1 权重等价于 softmax(top1)。

为什么这样写：
- router 通常是 MoE 层的高频小计算，减少中间张量比复用通用算子更重要。
- correction bias 放在 softcap 后，能在不改变 softcap 限幅语义的情况下做推理期负载修正。

不变量与失败模式：
- `topk` 必须在 `[1, num_experts]` 内；`hidden_dim` 必须与 router weight 的第二维一致。
- 若 `BLOCK_SIZE` 或 mask 处理错误，router weight 越界会直接污染 expert 选择。

Comment：
这是 fast path 的最小闭环：只产出后续 MoE 真正需要的 top-k id 与权重，不保留完整 logits。

### 1.2 topk>=2 通过排除 top1 得到第二专家

问题与约束：
- DeepSeek 等模型常用 top-k=2，需要第二个 expert 的 id 与 softmax 权重，同时不能重新计算完整 softmax。

设计选择：
- 在同一个 kernel 中把 top1 位置置为 `-inf` 后再 `argmax`，第二专家权重复用 top1 基准的 `invsumexp`。

Explain：
代码对 `topk >= 2` 单独处理：先用 `tl.where` 排除 top1，再求 top2。top2 权重通过 `exp(top2_v - top1_v) * invsumexp` 得到，避免额外除法和完整 softmax materialization。

来源：python/sglang/srt/layers/moe/router.py L77-L90

Code：

```python
if topk >= 2:
    top2 = tl.argmax(
        tl.where(
            tl.arange(0, num_experts) != top1, logits_softcapped, float("-inf")
        ),
        axis=0,
    )
    tl.store(topk_ids_ptr + pid * topk + 1, top2)
    top2_v = tl.sum(logits_softcapped * (tl.arange(0, num_experts) == top2), axis=0)
    tl.store(
        topk_weights_ptr + pid * topk + 1,
        tl.exp(top2_v - top1_v) * invsumexp,
    )
```

代码逻辑：
- `top2` 的候选集合等于所有 expert 减去 top1。
- `top2_v` 通过 one-hot mask 取回对应 logit。
- 权重复用第一段中已经算出的 `top1_v` 与 `invsumexp`。

为什么这样写：
- top-k=2 是常见路径，专门展开能避免慢速泛化 top-k。
- 该写法保持 top1/top2 权重来自同一个 softmax 归一化常数。

不变量与失败模式：
- 当 `topk >= 2` 时 `num_experts` 至少要有 2 个有效 expert。
- 若 top1 mask 没有生效，top2 可能重复 top1，后续 dispatch 会把同一 token 发送给重复 expert。

Comment：
`topk > 2` 后面进入更慢的泛化路径；这段主要优化实际模型里最常见的 top2。

### 1.3 Python wrapper 负责 buffer、launch config 与返回契约

问题与约束：
- Triton kernel 只处理 device 侧计算；Python 层还要保证输入形状、输出 dtype、launch grid 与 kernel meta 参数一致。

设计选择：
- `fused_moe_router_cudacore` 在进入 kernel 前断言 hidden dim，分配 `(bs, topk)` 输出，并按 hidden dim 推导 `BLOCK_SIZE/num_warps`。

Explain：
wrapper 从 `x/router_weight` 推出 batch、hidden dim 和 expert 数，输出 `float32` 权重与 `int32` expert id。launch 后只返回这两个张量，因此调用方无法依赖完整 router logits。

来源：python/sglang/srt/layers/moe/router.py L117-L156

Code：

```python
def fused_moe_router_cudacore(
    x: torch.Tensor,
    router_weight: torch.Tensor,
    topk: int,
    moe_softcapping: float,
    correction_bias: Optional[torch.Tensor] = None,
):
    assert len(x.shape) == 2 and x.shape[1] == router_weight.shape[1]
    bs, hidden_dim = x.shape
    num_experts = router_weight.shape[0]
    topk_weights = torch.empty((bs, topk), dtype=torch.float32, device=x.device)
    topk_ids = torch.empty((bs, topk), dtype=torch.int32, device=x.device)
    is_correction_bias = correction_bias is not None
    config = {
        "BLOCK_SIZE": triton.next_power_of_2(hidden_dim),
        "num_warps": max(
            min(triton.next_power_of_2(triton.cdiv(hidden_dim, 256)), max_warps), 4
        ),
    }
    fused_moe_router_cudacore_kernel[(bs,)](
        x,
        router_weight,
        topk_weights,
        topk_ids,
        correction_bias,
        is_correction_bias=is_correction_bias,
        num_experts=num_experts,
        topk=topk,
        moe_softcapping=moe_softcapping,
        moe_renormalize=False,
        hidden_dim=hidden_dim,
        **config,
    )
    return topk_weights, topk_ids
```

代码逻辑：
- shape assert 把 hidden dim 不一致的错误挡在 kernel launch 前。
- 输出 buffer dtype 是下游 dispatcher 和 fused MoE kernel 的输入契约。
- grid 设为 `(bs,)`，使每个 token 独立映射到一个 Triton program。

为什么这样写：
- 输出契约越窄，后续 `FusedMoE` 的 dispatch 输入越稳定。
- hidden dim 的 power-of-two block 便于 Triton 向量化和 mask 尾部。

不变量与失败模式：
- `x` 必须是二维 token×hidden；router weight 必须是 expert×hidden。
- `correction_bias` 若非空，长度必须覆盖全部 expert，否则 kernel 内加载会越界或读错。

Comment：
这一层是 Python 与 Triton 的边界：性能细节在 kernel，形状和资源配置在 wrapper。

## 2. TopK：路由算法选择与后处理

### 2.1 select_experts 先展开配置并接入 EPLB 输入变换

问题与约束：
- SGLang 需要支持 grouped top-k、torch native top-k、biased top-k、custom routing、EPLB remap、共享 expert 等多种模型与后端组合。

设计选择：
- `select_experts` 先把 `TopKConfig` 展开为局部变量，再调用 `expert_location_dispatch.transform_select_experts_inputs` 处理 EPLB 对 logits/bias 的输入变换。

Explain：
函数签名显式接收 `hidden_states/router_logits/topk_config`，并允许传入 `layer_id`、非 padding token 数、expert location 信息。EPLB 的输入变换在选择算法前执行，因此后续所有 top-k 分支都使用同一套变换后的 logits 和 bias。

来源：python/sglang/srt/layers/moe/topk.py L1876-L1912

Code：

```python
def select_experts(
    hidden_states: torch.Tensor,
    router_logits: torch.Tensor,
    topk_config: TopKConfig,
    *,
    layer_id: Optional[int] = None,
    num_token_non_padded: Optional[torch.Tensor] = None,
    expert_location_dispatch_info: Optional[ExpertLocationDispatchInfo] = None,
) -> StandardTopKOutput:
    top_k = topk_config.top_k
    use_grouped_topk = topk_config.use_grouped_topk
    correction_bias = topk_config.correction_bias
    torch_native = topk_config.torch_native
    scoring_func = topk_config.scoring_func
    (
        router_logits,
        correction_bias,
    ) = expert_location_dispatch.transform_select_experts_inputs(
        router_logits=router_logits,
        correction_bias=correction_bias,
        info=expert_location_dispatch_info,
    )
```

代码逻辑：
- `TopKConfig` 统一承载 top-k 数、分组参数、bias、scoring 函数和缩放策略。
- `expert_location_dispatch_info` 只在需要 EPLB 时参与输入变换。
- `packed_topk` 初始为 `None`，表示默认返回普通 `StandardTopKOutput`。

为什么这样写：
- 分支前集中展开配置，能让后续路径保持简单条件判断。
- EPLB 输入变换越早执行，越不容易出现不同 top-k 分支对负载均衡语义不一致。

不变量与失败模式：
- `TopKConfig` 中的 grouped top-k 参数必须成组出现；缺失会在后续 assert 中失败。
- 若 EPLB metadata 与当前 layer 不匹配，top-k 可能基于错误的 logical expert 空间。

Comment：
`select_experts` 是 MoE router 的策略层：它不执行 GEMM，只决定哪些 token 去哪些 expert。

### 2.2 TopK 分支按模型语义和后端能力选择实现

问题与约束：
- 不同模型使用不同 scoring 与 top-k 规则；不同后端又要求不同输入形态，不能只靠一个通用 top-k 覆盖所有情况。

设计选择：
- 按 grouped、torch native、sqrtsoftplus biased、FlashInfer raw-logits softmax 等条件分支选择具体实现。

Explain：
DeepSeek 系列 grouped top-k 先走 `grouped_topk/biased_grouped_topk`；`torch_native` 分支走原生 top-k；`sqrtsoftplus` 使用 biased top-k；FlashInfer TRT-LLM routed 后端在 softmax 且无 correction bias 时使用 raw-logits top-k。

来源：python/sglang/srt/layers/moe/topk.py L1920-L1993

Code：

```python
if use_grouped_topk:
    if correction_bias is None:
        topk_weights, topk_ids = grouped_topk(
            hidden_states=hidden_states,
            gating_output=router_logits,
            topk=num_routed_topk if _use_aiter else top_k,
            renormalize=renormalize,
            num_expert_group=num_expert_group,
            topk_group=topk_group,
        )
    else:
        topk_weights, topk_ids = biased_grouped_topk(
            hidden_states=hidden_states,
            gating_output=router_logits,
            correction_bias=correction_bias,
            topk=num_routed_topk if _use_aiter else top_k,
            renormalize=renormalize,
            num_expert_group=num_expert_group,
            topk_group=topk_group,
        )
elif torch_native and custom_routing_function is None:
    topk_weights, topk_ids = fused_topk_native(
        hidden_states=hidden_states,
        gating_output=router_logits,
        topk=num_routed_topk if _use_aiter else top_k,
        renormalize=renormalize,
        correction_bias=correction_bias,
        scoring_func=scoring_func,
    )
elif custom_routing_function is None:
    if scoring_func == "sqrtsoftplus":
        topk_weights, topk_ids = _biased_topk(...)
    elif (
        get_moe_runner_backend().is_flashinfer_trtllm_routed()
        and scoring_func == "softmax"
        and correction_bias is None
    ):
        topk_weights, topk_ids = fused_topk_softmax_torch_raw_logits(...)
```

代码逻辑：
- `num_routed_topk` 会排除 fused shared experts，AITER 路径再决定传入 routed top-k 还是原始 top-k。
- grouped top-k 对 correction bias 有单独实现。
- FlashInfer routed 路径避免提前 softmax，保留 raw logits 语义。

为什么这样写：
- MoE 的 top-k 不只是排序问题，还包含模型约定、硬件后端和负载均衡语义。
- 分支显式化使不支持的组合能通过 assert 或条件提前暴露。

不变量与失败模式：
- grouped top-k 必须提供 `topk_group/num_expert_group`。
- `torch_native` 分支暂不支持 `num_token_non_padded`，误用会触发 assert。
- correction bias 与 raw-logits FlashInfer 路径不兼容时会落入其他实现。

Comment：
这段是 TopK 的兼容矩阵，阅读时要把它理解成策略路由，而不是单一算法实现。

### 2.3 后处理同时服务 dispatcher 与 EPLB recorder

问题与约束：
- top-k 结果既要给 dispatcher 使用，也要给 EPLB 统计负载；两者对 logical/physical expert id 的需求不同。

设计选择：
- `_post_process_topk_ids` 返回 dispatcher 用的 `topk_ids/topk_weights`，同时返回 recorder 使用的 `recorder_topk_ids`。

Explain：
后处理阶段处理 padding 区域、EPLB logical→physical 映射和 benchmark 模拟策略。随后全局 expert distribution recorder 记录 `recorder_topk_ids`，函数最终返回 `StandardTopKOutput` 或 packed 变体。

来源：python/sglang/srt/layers/moe/topk.py L2091-L2111

Code：

```python
topk_ids, topk_weights, recorder_topk_ids = _post_process_topk_ids(
    topk_ids=topk_ids,
    topk_weights=topk_weights,
    topk_config=topk_config,
    router_logits=router_logits,
    num_token_non_padded=num_token_non_padded,
    layer_id=layer_id,
    expert_location_dispatch_info=expert_location_dispatch_info,
)

get_global_expert_distribution_recorder().on_select_experts(
    topk_ids=recorder_topk_ids
)

if packed_topk is not None:
    return StandardTopKOutputPacked(topk_weights, topk_ids, router_logits, packed_topk)
return StandardTopKOutput(topk_weights, topk_ids, router_logits)
```

代码逻辑：
- dispatcher 使用后处理后的 `topk_ids/topk_weights`。
- recorder 单独使用 `recorder_topk_ids`，避免统计口径被物理 dispatch id 污染。
- packed top-k 是额外优化路径，默认仍返回标准结构。

为什么这样写：
- 推理执行和负载观测是两个消费者，拆开 id 口径可以避免 EPLB 统计误读。
- recorder 调用放在返回前，保证所有 top-k 分支都会经过同一统计出口。

不变量与失败模式：
- `_post_process_topk_ids` 必须返回形状一致的三元组。
- 如果 `layer_id` 缺失但后处理需要按层 remap，EPLB 统计或 dispatch 都可能失真。

Comment：
这段把 TopK 从“选 expert”扩展成“产出可执行、可观测的路由结果”。

## 3. FusedMoE：dispatch/core/combine 主路径

### 3.1 forward 先处理平台和 CUDA Graph 特化路径

问题与约束：
- MoE 层要兼容 Ascend fuseep、piecewise CUDA Graph、bypassed top-k 和普通 eager 路径。

设计选择：
- `forward` 只做模式分发：Ascend 直接进 fuseep；CUDA Graph 下按 `TopKOutput` 格式走注册 op；其他情况进入 `forward_impl`。

Explain：
`TopKOutputChecker` 决定 top-k 输出是标准格式还是 bypassed 格式。CUDA Graph 路径需要整层有 torch library op 注册，无法处理的格式退回 `forward_impl`。

来源：python/sglang/srt/layers/moe/fused_moe_triton/layer.py L1102-L1132

Code：

```python
def forward(self, hidden_states: torch.Tensor, topk_output: TopKOutput):
    if self._use_ascend_fuseep:
        from sglang.srt.hardware_backend.npu.moe.fuseep import forward_fuseep

        return forward_fuseep(self, hidden_states, topk_output)
    if is_in_tc_piecewise_cuda_graph():
        if TopKOutputChecker.format_is_standard(topk_output):
            return moe_forward_piecewise_cuda_graph_impl(...)
        elif TopKOutputChecker.format_is_bypassed(topk_output):
            return fused_moe_bypassed_piecewise_cuda_graph_impl(...)
        else:
            return self.forward_impl(hidden_states, topk_output)
    else:
        return self.forward_impl(hidden_states, topk_output)
```

代码逻辑：
- Ascend 分支在最前，直接交给硬件后端实现。
- CUDA Graph 分支根据 top-k 输出格式选择已注册的图内 op。
- 非图模式和不支持格式都使用统一 `forward_impl`。

为什么这样写：
- 把平台特化留在 `forward`，可以让 `forward_impl` 保持通用三阶段语义。
- CUDA Graph 路径依赖稳定 op 注册，不能把任意 Python 分支混进 capture。

不变量与失败模式：
- `TopKOutputChecker` 必须准确区分输出格式。
- 若 bypassed top-k 的 config 字段缺失，CUDA Graph 特化调用会在参数展开处失败。

Comment：
这一层不解释 MoE 算法本身，而是把运行时模式收敛到可执行路径。

### 3.2 forward_impl 固定为 dispatch → core → combine

问题与约束：
- MoE 执行既涉及 token permutation/通信，也涉及 expert GEMM，还要把结果还原到原 token 顺序。

设计选择：
- `forward_impl` 固定三阶段：dispatcher dispatch，`run_moe_core` 做专家计算，dispatcher combine，再按 TP/EP 需要 all-reduce。

Explain：
函数先保存原 hidden dim，避免 padding 后输出维度泄露给上层。dispatch 输出交给 MoE core，combine 后切回原 hidden dim；当 `reduce_results` 且 TP/EP 大于 1 时做 tensor parallel all-reduce。

来源：python/sglang/srt/layers/moe/fused_moe_triton/layer.py L1134-L1159

Code：

```python
def forward_impl(self, hidden_states: torch.Tensor, topk_output: TopKOutput):
    origin_hidden_states_dim = hidden_states.shape[-1]
    assert self.quant_method is not None
    dispatch_output = self.dispatcher.dispatch(
        hidden_states=hidden_states, topk_output=topk_output
    )
    combine_input = self.run_moe_core(
        dispatch_output=dispatch_output,
    )
    with use_symmetric_memory(get_tp_group(), disabled=not is_allocation_symmetric()):
        final_hidden_states = self.dispatcher.combine(combine_input=combine_input)
        final_hidden_states = final_hidden_states[..., :origin_hidden_states_dim].contiguous()
    if self.reduce_results and (self.moe_tp_size > 1 or self.moe_ep_size > 1):
        final_hidden_states = tensor_model_parallel_all_reduce(final_hidden_states)
    return final_hidden_states
```

代码逻辑：
- `dispatcher.dispatch` 负责把 token 按 expert 和 rank 整理成 core 可消费格式。
- `run_moe_core` 返回 combine 所需输入，而不是直接返回最终 hidden states。
- symmetric memory 只包住 combine 与裁剪，减少跨 rank memory 语义扩散。

为什么这样写：
- dispatch/core/combine 分离后，A2A 后端和 GEMM 后端可以独立替换。
- 输出裁剪放在 combine 后，允许内部 kernel 做 padding 对齐而不改变上层 shape。

不变量与失败模式：
- `quant_method` 必须存在；否则无法选择 MoE runner。
- dispatcher 的 dispatch output 与 combine input 格式必须匹配同一个后端。

Comment：
这是读 MoE 执行路径最重要的骨架：通信与计算解耦，但张量契约必须闭合。

### 3.3 run_moe_core 把专家计算委托给 quant_method

问题与约束：
- MoE GEMM 会随量化格式、权重布局、后端 runner 而变化，`FusedMoE` 不适合内置所有 kernel 分支。

设计选择：
- `run_moe_core` 只调用 `self.quant_method.apply(layer=self, dispatch_output=...)`，由量化方法绑定实际 runner。

Explain：
这段代码很短，但边界清楚：`FusedMoE` 提供 layer 上下文和 dispatch output，具体如何运行 W13/W2、FP8/INT4/普通 Triton kernel 由 quant method 决定。

来源：python/sglang/srt/layers/moe/fused_moe_triton/layer.py L1178-L1183

Code：

```python
def run_moe_core(self, dispatch_output: DispatchOutput) -> CombineInput:
    return self.quant_method.apply(
        layer=self,
        dispatch_output=dispatch_output,
    )
```

代码逻辑：
- 输入是 dispatcher 产物，输出是 combine input。
- `layer=self` 让 quant method 能访问 expert weights、bias、runner config 和并行配置。

为什么这样写：
- 量化和 kernel 选择是横切关注点，放在 quant method 中比在 MoE 层堆分支更可维护。
- dispatch/combine 不需要知道 GEMM 细节，只关心输入输出契约。

不变量与失败模式：
- quant method 必须返回 dispatcher 能识别的 `CombineInput`。
- 若 quant method 与 dispatcher 后端不匹配，错误通常会在 combine 格式检查或张量 shape 处暴露。

Comment：
MoE 的核心计算没有直接写在 `FusedMoE` 里，而是通过 quant method 做一次后端分发。

## 4. Dispatcher：hook、DeepEP 与格式边界

### 4.1 BaseDispatcher 用函数替换注入 pre/post hook

问题与约束：
- EPLB、观测和 overlap 需要在 dispatch/combine 前后插入逻辑，但不应让每个 dispatcher 实现都手写同一套 hook 调用。

设计选择：
- `BaseDispatcher` 懒创建 hook 容器，并把原始 `dispatch/combine` 保存到 `_original_*` 后替换为带 hook 的 wrapper。

Explain：
注册第一个 hook 时，dispatcher 会把当前方法保存为原始函数，然后把公开方法替换为 `_dispatch_with_hook` 或 `_combine_with_hook`。hook 可修改输入或输出，返回 `None` 时保持原值。

来源：python/sglang/srt/layers/moe/token_dispatcher/base.py L261-L359

Code：

```python
class BaseDispatcher(ABC):
    def __init__(self):
        self._pre_dispatch_hooks: Optional[_PreDispatchHooks] = None
        self._post_dispatch_hooks: Optional[_PostDispatchHooks] = None
        self._pre_combine_hooks: Optional[_PreCombineHooks] = None
        self._post_combine_hooks: Optional[_PostCombineHooks] = None
        self._original_dispatch_func: Optional[Callable] = None
        self._original_combine_func: Optional[Callable] = None

    def _dispatch_with_hook(self, hidden_states, topk_output) -> DispatchOutput:
        if self._pre_dispatch_hooks is not None:
            hidden_states, topk_output = self._pre_dispatch_hooks(
                self, hidden_states, topk_output
            )
        dispatch_output = self._original_dispatch_func(
            hidden_states=hidden_states, topk_output=topk_output
        )
        if self._post_dispatch_hooks is not None:
            dispatch_output = self._post_dispatch_hooks(self, dispatch_output)
        return dispatch_output

    def register_pre_dispatch_hook(self, hook: Callable) -> _RemovableDispatcherHandle:
        if self._pre_dispatch_hooks is None:
            self._pre_dispatch_hooks = _PreDispatchHooks()
            self._override_dispatch_func()
        handle = self._pre_dispatch_hooks.register_hook(hook)
        return handle
```

代码逻辑：
- hook 容器为空时不包裹方法，避免无 hook 场景额外开销。
- 注册 hook 后，公开方法仍叫 `dispatch/combine`，调用方无感。
- 返回 handle 允许外部取消 hook。

为什么这样写：
- dispatcher 子类只实现核心通信逻辑，扩展逻辑通过基类统一注入。
- 运行期 hook 比继承层级更适合 EPLB 这种可选能力。

不变量与失败模式：
- `_original_dispatch_func` 只能保存一次；重复覆盖会导致 wrapper 套 wrapper。
- hook 修改张量 shape 或格式时必须保持 dispatcher 契约，否则后续 core/combine 会失败。

Comment：
这就是旧笔记里提到的 hook 机制真实位置：不是 combine format enum，而是 `BaseDispatcher` 的方法替换。

### 4.2 DeepEPDispatcher 把 dispatch/combine 拆成 A/B 两阶段

问题与约束：
- DeepEP 通信涉及异步 send/recv、normal/low-latency 两种模式和可选 hook，单个同步函数难以表达 overlap。

设计选择：
- 对外 `dispatch` 调用 `dispatch_a`、执行 DeepEP hook、再 `dispatch_b`；combine 同样拆为 `combine_a/combine_b`。

Explain：
初始化时根据 `DeepEPMode` 构造 normal 或 low-latency 实现。stage 状态机约束调用顺序，intermediate state 在 A 阶段保存，B 阶段读取后删除。

来源：python/sglang/srt/layers/moe/token_dispatcher/deepep.py L841-L951

Code：

```python
class DeepEPDispatcher(BaseDispatcher):
    def __init__(..., deepep_mode: DeepEPMode = DeepEPMode.AUTO, ...):
        super().__init__()
        if self.deepep_mode.enable_low_latency():
            self._low_latency_dispatcher = _DeepEPDispatcherImplLowLatency(...)
        if self.deepep_mode.enable_normal():
            self._normal_dispatcher = _DeepEPDispatcherImplNormal(...)
        self._stage = _Stage.INITIAL
        self._deepep_dispatch_hooks = DeepEPPDispatchHooks()

    def dispatch(self, hidden_states: torch.Tensor, topk_output: TopKOutput) -> DispatchOutput:
        self.dispatch_a(hidden_states, topk_output)
        if self._deepep_dispatch_hooks is not None:
            self._deepep_dispatch_hooks(self)
        ret = self.dispatch_b()
        return ret

    def dispatch_a(self, hidden_states: torch.Tensor, topk_output: TopKOutput):
        self._update_stage(_Stage.INITIAL, _Stage.AFTER_DISPATCH_A)
        inner_state = self._get_impl().dispatch_a(
            hidden_states=hidden_states,
            topk_output=topk_output,
        )
        self._dispatch_intermediate_state = inner_state

    def combine(self, combine_input: CombineInput) -> torch.Tensor:
        self.combine_a(combine_input)
        ret = self.combine_b()
        return ret
```

代码逻辑：
- `dispatch_a` 进入后端实现并保存中间状态。
- DeepEP hook 在 A/B 之间执行，可读取 dispatch 统计。
- `dispatch_b/combine_b` 读取中间状态后删除，防止跨 batch 复用旧状态。

为什么这样写：
- A/B 拆分为 overlap 和观测留出插入点。
- normal/low-latency 实现共享外层状态机，减少调用方分支。

不变量与失败模式：
- stage 顺序必须严格匹配 INITIAL→AFTER_DISPATCH_A→AFTER_DISPATCH_B→AFTER_COMBINE_A→INITIAL。
- 如果 B 阶段未调用，中间状态会滞留；如果重复调用，已删除状态会缺失。

Comment：
DeepEP dispatcher 的关键不是单纯 all-to-all，而是把通信生命周期拆成可 overlap、可观测的阶段。

## 5. EPLB：logical→physical 映射与负载记录

### 5.1 ExpertLocationDispatchInfo 封装每层 expert 位置元数据

问题与约束：
- EPLB 会改变 logical expert 与 physical slot 的对应关系；router 与 dispatcher 需要按 layer 读取同一份位置元数据。

设计选择：
- 用 `ExpertLocationDispatchInfo` 保存 dispatch algorithm、logical→rank physical map、logical→all physical map 和 physical expert 数。

Explain：
`init_new(layer_id)` 从全局 server args 与 expert location metadata 读取当前层映射。如果没有配置 `ep_dispatch_algorithm`，直接返回 `None`，让普通路径不承担 EPLB 成本。

来源：python/sglang/srt/eplb/expert_location_dispatch.py L27-L64

Code：

```python
@dataclass
class ExpertLocationDispatchInfo:
    ep_dispatch_algorithm: Literal["static", "random"]
    partial_logical_to_rank_dispatch_physical_map: Optional[torch.Tensor]
    partial_logical_to_all_physical_map: torch.Tensor
    partial_logical_to_all_physical_map_num_valid: torch.Tensor
    num_physical_experts: int

    @classmethod
    def init_new(cls, layer_id: int):
        ep_dispatch_algorithm = get_global_server_args().ep_dispatch_algorithm
        expert_location_metadata = get_global_expert_location_metadata()
        assert expert_location_metadata is not None
        if ep_dispatch_algorithm is None:
            return None
        return cls(
            ep_dispatch_algorithm=ep_dispatch_algorithm,
            partial_logical_to_rank_dispatch_physical_map=(
                expert_location_metadata.logical_to_rank_dispatch_physical_map[layer_id, :]
                if expert_location_metadata.logical_to_rank_dispatch_physical_map is not None
                else None
            ),
            partial_logical_to_all_physical_map=expert_location_metadata.logical_to_all_physical_map[layer_id, :],
            partial_logical_to_all_physical_map_num_valid=expert_location_metadata.logical_to_all_physical_map_num_valid[layer_id, :],
            num_physical_experts=expert_location_metadata.num_physical_experts,
        )
```

代码逻辑：
- layer id 索引出当前层的映射表。
- rank-local 和 all-physical 两套映射同时保留，服务不同算法。
- `None` 表示完全绕过 EPLB remap。

为什么这样写：
- 元数据封装成对象后，TopK 和 dispatch 后处理只需要传一个 `info`。
- 以 layer 为单位读取，符合 MoE 每层 expert 分布可能不同的事实。

不变量与失败模式：
- 开启 EPLB 时全局 metadata 必须存在。
- metadata 的 layer 维度必须覆盖当前 layer，否则索引会越界。

Comment：
这个 dataclass 是 EPLB 进入 router/dispatcher 路径的载体。

### 5.2 topk_ids_logical_to_physical 选择映射算法

问题与约束：
- router 输出的是 logical expert id，但 dispatch 需要 physical expert slot；不同 rebalance 策略需要不同映射方式。

设计选择：
- 用 `ep_dispatch_algorithm` 在 static、dynamic/fake、lp 三类映射之间分发。

Explain：
没有 `info` 时直接返回原 `topk_ids`。static 路径查 rank-local map；dynamic/fake 路径从每个 logical expert 的多个 physical 候选中随机选择；lp 路径要求额外概率矩阵并走 CUDA solver。

来源：python/sglang/srt/eplb/expert_location_dispatch.py L79-L127

Code：

```python
def topk_ids_logical_to_physical(
    topk_ids: torch.Tensor,
    info: Optional[ExpertLocationDispatchInfo],
    log2phy_prob: Optional[torch.Tensor] = None,
) -> torch.Tensor:
    if info is None:
        return topk_ids
    if info.ep_dispatch_algorithm == "static":
        return _topk_ids_logical_to_physical_static(topk_ids, info)
    if info.ep_dispatch_algorithm in ["dynamic", "fake"]:
        return _topk_ids_logical_to_physical_dynamic(topk_ids, info)
    if info.ep_dispatch_algorithm == "lp":
        if log2phy_prob is None:
            raise RuntimeError("ep_dispatch_algorithm='lp' but log2phy_prob is None")
        return _topk_ids_logical_to_physical_probability(topk_ids, info, log2phy_prob)
    raise NotImplementedError(f"Unknown algorithm {info.ep_dispatch_algorithm}")

def _topk_ids_logical_to_physical_static(topk_ids, info):
    physical_topk_ids = info.partial_logical_to_rank_dispatch_physical_map[topk_ids]
    if _is_hip:
        physical_topk_ids = physical_topk_ids.to(topk_ids.dtype)
    return physical_topk_ids
```

代码逻辑：
- `info is None` 是普通 MoE 的零成本路径。
- static 直接 table lookup。
- dynamic 先 flatten，再用随机 index 选择 physical map，最后恢复原 shape。

为什么这样写：
- logical id 到 physical slot 的转换必须在 dispatch 前完成，否则 token 会发到错误权重。
- 不同算法保持同一函数出口，TopK 后处理不需要知道具体映射细节。

不变量与失败模式：
- static 路径要求 `partial_logical_to_rank_dispatch_physical_map` 非空。
- lp 路径必须提供 `log2phy_prob` 且 tensor 在 CUDA 上，否则 runtime 会拒绝。

Comment：
EPLB 的核心动作不是改变 router logits，而是把选出的 logical expert 转成当前部署下可执行的 physical expert。

### 5.3 ExpertDistributionRecorder 用 hook 门控记录

问题与约束：
- 负载记录不能无条件执行，否则会污染普通推理开销；但 CUDA graph capture 期间仍要保持必要统计。

设计选择：
- recorder 的 `on_select_experts` 统一转发到 `_on_hook`，由 `_recording`、capture 状态和 gatherer key 决定是否写入。

Explain：
`_on_hook` 先检查全局禁用、是否 recording 或当前 stream 正在 capture，再根据 debug name 找 single-pass gatherer 并传入当前 layer id。`start_record/stop_record/dump_record` 管理一次记录周期。

来源：python/sglang/srt/eplb/expert_distribution.py L209-L280

Code：

```python
def on_select_experts(self, topk_ids: torch.Tensor):
    self._on_hook("on_select_experts", topk_ids=topk_ids)

def _on_hook(self, hook_name: str, **kwargs):
    if self._disable_all:
        return
    if not (
        self._recording or torch.get_device_module().is_current_stream_capturing()
    ):
        return
    gatherer = self._single_pass_gatherers[
        self._accumulator.get_single_pass_gatherer_key(
            self._current_debug_name.value
        )
    ]
    getattr(gatherer, hook_name)(layer_idx=self._current_layer_idx.value, **kwargs)

def start_record(self):
    if self._recording:
        logger.warning("SGLang server is already recording expert ids...")
    self._reset()
    self._recording = True

def dump_record(self, output_mode: _OutputMode = "file"):
    output = self._accumulator.dump(output_mode=output_mode)
    self._reset()
    return output
```

代码逻辑：
- `on_select_experts` 不直接写 buffer，只派发 hook。
- `_current_layer_idx` 在 hook 调用时补充 layer 维度。
- dump 后 reset，避免下一轮记录混入旧数据。

为什么这样写：
- 记录是调试与 EPLB 输入，不应成为普通推理的固定成本。
- hook 名称派发让 select experts、DeepEP normal、low-latency 等事件复用同一 recorder。

不变量与失败模式：
- 调用 hook 时必须已经设置当前 layer/debug name。
- 如果记录周期未 start，普通 eager 推理下 `on_select_experts` 会直接返回。

Comment：
负载统计链路是按需开启的观测面，不是 router 必经的计算面。

### 5.4 single-pass gatherer 按 layer 写入 topk ids

问题与约束：
- EPLB 后续聚合需要知道每层每个 token 的 expert 选择，而 TopK 只在当前层局部产生结果。

设计选择：
- gatherer 在 `on_select_experts` 中把当前层 `topk_ids` 写入预分配的 layer×token×k buffer。

Explain：
写入切片使用 `topk_ids.shape[0]` 和 `topk_ids.shape[1]` 限定 token 与 k 维度，只覆盖当前 batch 的有效区域。

来源：python/sglang/srt/eplb/expert_distribution.py L410-L413


不变量与失败模式：
- 当前 batch token 数和 top-k 数不能超过 buffer 预设容量。
- 若 layer idx 错误，统计会落到错误层，rebalance 决策会偏移。
