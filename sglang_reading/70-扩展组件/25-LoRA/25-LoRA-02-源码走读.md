# LoRA · 源码走读

> 走读主线：`LoRARegistry` 在 tokenizer 侧维护可见 adapter 和引用计数；`LoRAManager` 在 scheduler/model 侧加载 adapter、替换模块并维护 GPU 槽位；`LoRAMemoryPool` 把不同 adapter 权重装入固定 buffer；backend 根据 batch 的 adapter 槽位构造 segmented metadata，最终由 dense/embedding/MoE kernel 读同一套 buffer。

---

## 1. 控制面：manager、registry 与动态加载

### 1.1 LoRAManager 把静态配置、backend 和可变状态分开

问题与约束：
- LoRA serving 同时依赖 server args、base model config、TP rank、backend 类型和动态 adapter 状态；这些状态如果混在构造函数里，会让运行时 load/unload 难以保持一致。

设计选择：
- 构造函数只固化配置和 backend，再把可变状态交给 `init_state` 初始化。

Explain：
`LoRAManager.__init__` 先规范化 `base_hf_config`，保存 max slot、dtype、device、TP 信息和 MoE 相关开关，再通过 `get_backend_from_name` 创建 backend。真正的 adapter、shape、module wrapping、memory pool 初始化被延后到 `init_state`。

来源：python/sglang/srt/lora/lora_manager.py L57-L112

Code：

```python
self.max_loras_per_batch = max_loras_per_batch
self.load_config = load_config
self.dtype = dtype
self.device = device
self.tp_size = tp_size
self.tp_rank = tp_rank
self.eviction_policy = server_args.lora_eviction_policy
self.lora_use_virtual_experts = server_args.lora_use_virtual_experts

backend_type = get_backend_from_name(lora_backend)
self.lora_backend: BaseLoRABackend = backend_type(
    max_loras_per_batch=max_loras_per_batch,
    device=self.device,
    server_args=server_args,
)

self.init_state(
    max_lora_rank=max_lora_rank,
    target_modules=target_modules,
    lora_paths=lora_paths,
)
```

代码逻辑：
- 保存 serving 时不会随 batch 改变的配置。
- 根据 backend name 创建 `BaseLoRABackend` 子类实例。
- 调用 `init_state` 初始化 adapter 和模型层绑定。

为什么这样写：
- backend 的类型决定后续 batch metadata 和 kernel 调度方式，必须先于 module 绑定确定。
- 可变状态集中在 `init_state`，便于初始 `--lora-paths` 和后续动态加载共享同一套路径。

不变量与失败模式：
- `lora_backend` 必须能解析为 backend class。
- `max_loras_per_batch` 是所有 memory pool 和 batch metadata 的槽位上限，后续不能随请求变化。

Comment：
这段把 LoRA serving 的控制面分成配置层、backend 层和运行时状态层。

### 1.2 init_state 固定 adapter、shape、module、pool 的初始化顺序

问题与约束：
- 初始 adapter 可能用于推断 target modules、max rank 和 extra vocab；但 module wrapping 和 memory pool 分配又依赖这些推断结果。

设计选择：
- `init_state` 先加载初始 adapter/config，再推断 shape，随后包装模型模块、创建 memory pool，最后把 buffer 指针写回各 LoRA layer。

Explain：
初始化顺序是源码中的关键不变量：`init_lora_adapters` 让 adapter config 可见，`init_lora_shapes` 决定 target/rank/token size，shared-outer MoE 格式在包装模块前确定，然后 `init_lora_modules`、`init_memory_pool`、`update_lora_info` 串起模型和 buffer。

来源：python/sglang/srt/lora/lora_manager.py L435-L470

Code：

```python
self.init_lora_adapters(lora_paths)
self.init_lora_shapes(
    max_lora_rank=max_lora_rank,
    target_modules=target_modules,
)

if self._experts_shared_outer_override is not None:
    self.experts_shared_outer_loras = self._experts_shared_outer_override
else:
    self.experts_shared_outer_loras = self._detect_shared_outer_loras()

self.init_lora_modules()
self.init_memory_pool()
self.update_lora_info()
```

代码逻辑：
- 没有初始 LoRA 时，必须显式提供 `max_lora_rank` 和 `target_modules`。
- 先解析 adapter，再用 adapter 或 CLI 推断 shape。
- module wrapping 发生在 memory pool 创建前。
- `update_lora_info` 把 memory pool 的 tensor 绑定到 wrapped layer。

为什么这样写：
- wrapped layer 需要知道 target module 集合；memory pool 需要知道 wrapped layer 的 A/B buffer 形状。
- shared-outer MoE 影响 buffer 维度，必须在分配前确定。

不变量与失败模式：
- 如果没有 `lora_paths` 且缺少 rank/target modules，初始化会 assert。
- 混合 shared-outer 与 per-expert MoE adapter 会在检测阶段失败。

Comment：
`init_state` 是 LoRA serving 从配置进入可执行模型的总装配线。

### 1.3 LoRARef 用稳定 ID 支撑多节点一致性

问题与约束：
- LoRA 名称和路径可能重复或在不同进程中独立解析；多节点请求广播时，如果每个节点生成不同 id，就无法用同一个请求 id 找到同一个 adapter。

设计选择：
- `LoRARef` 把 `lora_id/lora_name/lora_path/pinned` 封装成不可变记录；初始 `--lora-paths` 可用 `uuid5` 从 name/path 生成 deterministic id。

Explain：
动态注册默认可以用 `uuid4` 生成新 id，但多节点初始加载需要稳定 id。`__post_init__` 明确拒绝 `None` id，防止 base model 的 `None` uid 与 adapter uid 混淆。

来源：python/sglang/srt/lora/lora_registry.py L28-L63

Code：

```python
class LoRARef(msgspec.Struct, frozen=True, array_like=True):
    lora_id: str = msgspec.field(default_factory=lambda: uuid4().hex)
    lora_name: Optional[str] = None
    lora_path: Optional[str] = None
    pinned: Optional[bool] = None

    def __post_init__(self):
        if self.lora_id is None:
            raise ValueError("lora_id cannot be None")

    @staticmethod
    def deterministic_id(lora_name: str, lora_path: str) -> str:
        return uuid5(NAMESPACE_URL, f"{lora_name}\0{lora_path}").hex
```

代码逻辑：
- `lora_id` 默认随机生成。
- 初始 adapter 可按 name/path 生成稳定 id。
- `pinned` 与后续 memory pool eviction 规则相连。

为什么这样写：
- 请求运行时只应携带稳定 adapter id，不应依赖可变名称或路径。
- `None` 被用于 base model 槽位，adapter id 不能占用这个语义。

不变量与失败模式：
- 同一 name/path 在多节点上必须生成同一 deterministic id。
- name/path 任一变化都会得到不同 id，缓存和 registry 会视为不同 adapter。

Comment：
`LoRARef` 是 tokenizer 侧 registry 和 scheduler 侧 manager 之间的 adapter 身份契约。

### 1.4 LoRARegistry 维护可见 adapter 与请求引用计数

问题与约束：
- tokenizer manager 能收到并发推理请求和动态 load/unload 请求；unload 时不能让仍在使用的 adapter 被提前删除。

设计选择：
- registry 用 RWLock 保护注册表更新，用 `ConcurrentCounter` 记录 adapter 正在被多少请求引用。

Explain：
注册表是 tokenizer manager 侧的 single source of truth。`register/unregister` 拿 writer lock，`acquire` 在 reader lock 下按 name 查 `LoRARef` 并递增 counter；命中时还会 `move_to_end`，给 LRU 选择提供最近使用信息。

来源：python/sglang/srt/lora/lora_registry.py L65-L160

Code：

```python
self._registry_lock = RWLock()
self._registry: OrderedDict[str, LoRARef] = OrderedDict()
self._counters: Dict[str, ConcurrentCounter] = {}

async def register(self, lora_ref: LoRARef):
    async with self._registry_lock.writer_lock:
        return self._register_adapter(lora_ref)

async def acquire(self, lora_name: Union[str, List[str]]):
    async with self._registry_lock.reader_lock:
        def _lookup(name: str):
            lora_ref = self._registry.get(name, None)
            if lora_ref:
                self._registry.move_to_end(name)
                return lora_ref.lora_id
            return None
```

代码逻辑：
- registry key 是 `lora_name`，请求实际使用 `lora_id`。
- acquire 支持单个 name 或 name list。
- LRU 顺序通过 `OrderedDict.move_to_end` 更新。
- counter 与 registry 并列维护。

为什么这样写：
- load/unload 是低频写操作，inference acquire/release 是高频读操作，读写锁能减少互相阻塞。
- LRU 信息放在 registry 侧，能服务动态卸载策略。

不变量与失败模式：
- 注册同名 LoRA 会失败。
- 如果调用方 acquire 后不 release，unload 等待会被 counter 阻塞。

Comment：
Registry 解决“哪些 adapter 可见”和“哪些 adapter 仍被请求持有”两个问题。

### 1.5 release 与 wait_for_unload 形成安全卸载屏障

问题与约束：
- unregister 只能阻止新请求拿到 adapter，不能证明旧请求已经完成。

设计选择：
- 请求结束时调用 `release` 递减 counter；unload 流程在 unregister 后调用 `wait_for_unload` 等 counter 归零，再删除 counter。

Explain：
`wait_for_unload` 要求 adapter 已经不在 registry 中，但 counter 仍存在。这样新请求不会再 acquire，旧请求释放后 counter 才能归零，卸载变成两阶段一致性过程。

来源：python/sglang/srt/lora/lora_registry.py L167-L203

Code：

```python
async def release(self, lora_id: Union[str, List[str]]):
    async with self._registry_lock.reader_lock:
        if isinstance(lora_id, str):
            await self._counters[lora_id].decrement()
        elif isinstance(lora_id, list):
            await asyncio.gather(
                *[self._counters[id].decrement() for id in lora_id if id is not None]
            )

async def wait_for_unload(self, lora_id: str):
    assert lora_id not in self._registry
    assert lora_id in self._counters
    await self._counters[lora_id].wait_for_zero()
    del self._counters[lora_id]
```

代码逻辑：
- release 可处理单个 id 或 id list。
- `None` id 被跳过，因为它表示 base model。
- wait 阶段只等待 counter，不再访问 registry 项。

为什么这样写：
- unregister 后新请求不可见，counter 只代表历史请求。
- 删除 counter 放在 wait 之后，避免 release 找不到计数器。

不变量与失败模式：
- `wait_for_unload` 必须在 unregister 后调用。
- release 漏调会导致 unload 永远等待。

Comment：
这段是动态卸载避免“请求仍在跑但 adapter 已释放”的核心保护。

### 1.6 动态 load 在进入 memory pool 前做兼容性校验

问题与约束：
- 动态加载发生在服务运行中，不能让不兼容的 adapter 改写已有状态或污染 GPU buffer。

设计选择：
- `load_lora_adapter` 先构造 `LoRAConfig`，通过 `validate_new_adapter` 检查 vocab、DoRA、重名、memory pool 能力和 pinned 数量，再加载权重并登记引用。

Explain：
源码当前拒绝 added tokens 和 DoRA；若 adapter rank 超过 `max_lora_rank` 或 target modules 不在 pool 支持集合中，`memory_pool.can_support` 会阻止加载。pinned adapter 不能占满所有槽位，避免 base model 或 unpinned adapter 饥饿。

来源：python/sglang/srt/lora/lora_manager.py L154-L246

Code：

```python
new_adapter = LoRAConfig(
    lora_ref.lora_path,
    base_vocab_size=self.base_hf_config.vocab_size,
)
self.validate_new_adapter(new_adapter, lora_ref)
self.configs[lora_ref.lora_id] = new_adapter
self.load_lora_weights(lora_ref)
self.lora_refs[lora_ref.lora_id] = lora_ref
self.num_pinned_loras += int(lora_ref.pinned)
```

```python
if lora_config.lora_added_tokens_size > 0:
    raise ValueError(...)
if lora_config.use_dora:
    raise ValueError(...)
incompatible = memory_pool and not memory_pool.can_support(lora_config)
if incompatible:
    raise ValueError(...)
if lora_ref.pinned and self.num_pinned_loras >= self.max_loras_per_batch - 1:
    raise ValueError(...)
```

代码逻辑：
- 先读 config，不立即写入 GPU buffer。
- 校验通过后才保存 config、加载 CPU 权重、登记 ref。
- 所有异常都被封装成 `LoRAUpdateOutput(success=False)`。

为什么这样写：
- 动态 load 要保持失败原子性，不能部分修改可见 adapter 集。
- pinned 数量必须小于槽位上限，否则 eviction 没有可用候选。

不变量与失败模式：
- 同名 adapter 不能重复加载。
- 同路径不同名只 warning，仍可加载为不同 adapter。
- adapter 的 rank 和 target modules 必须匹配当前 memory pool。

Comment：
动态加载的关键不是读权重，而是先证明这个 adapter 能被当前 serving 实例承载。

### 1.7 unload 与 batch 校验保护槽位容量

问题与约束：
- 一个 batch 可能同时请求多个 LoRA；scheduler 必须在进入 forward 前确认 slot 数足够，且 pinned adapter 不会被误淘汰。

设计选择：
- unload 删除 CPU 侧 config/weights/ref 并维护 pinned 计数；`validate_lora_batch` 根据 batch 内 unique id、pinned 数量和 memory pool vacancy 判断是否可调度。

Explain：
`required_slots = len(lora_ids) - pinned_loras_in_batch`，可用槽位则是 `max_loras_per_batch - num_pinned_loras`。这把 pinned adapter 当作常驻占位，避免调度阶段低估实际槽位压力。

来源：python/sglang/srt/lora/lora_manager.py L248-L302

Code：

```python
def unload_lora_adapter(self, lora_ref: LoRARef) -> LoRAUpdateOutput:
    self.configs.pop(lora_ref.lora_id, None)
    self.loras.pop(lora_ref.lora_id, None)
    self.lora_refs.pop(lora_ref.lora_id, None)
    self.num_pinned_loras -= int(lora_ref.pinned)
    return self.create_lora_update_result(success=True)

required_slots = len(lora_ids) - pinned_loras_in_batch
mem_pool_vacancy = self.memory_pool.max_loras_per_batch - self.num_pinned_loras
return required_slots <= mem_pool_vacancy
```

代码逻辑：
- unload 只移除 manager CPU 侧记录。
- batch 校验先统计 unique LoRA ids。
- pinned adapter 计入已占用槽位，不参与普通 vacancy。

为什么这样写：
- 运行中 GPU buffer 的实际换入换出由 memory pool 在 batch 准备时处理。
- batch 校验提前失败，比进入 forward 后发现槽位不够更可控。

不变量与失败模式：
- `pinned_loras_in_batch` 不应超过全局 pinned 数量，源码用 assert 检查这个内部一致性。
- 如果 batch unique LoRA 数超过 `max_loras_per_batch`，调度必须拒绝或拆 batch。

Comment：
LoRA batch 的容量约束是“unique adapter 数”而不是请求数。

## 2. 模型改写与权重规范化

### 2.1 init_lora_shapes 从 CLI 或 adapter config 推断 target/rank

问题与约束：
- 用户可以显式给出 target modules，也可以让初始 adapter config 推断；同时 PEFT 的 `"all"`、`"all-linear"` 等 shorthand 不一定能直接映射到 SGLang 模块名。

设计选择：
- 显式 target modules 优先；`"all"` 通过扫描 base model 自动解析；adapter config 中的 target modules 必须是支持的 list 或可识别 shorthand。

Explain：
函数会把 target modules 规范化，并校验 adapter target 是否是 CLI target 的子集。`max_lora_rank` 可由 CLI 给出，否则从已加载 config 的 `r` 取最大值。extra vocab size 也从初始 adapter config 自动推断。

来源：python/sglang/srt/lora/lora_manager.py L530-L649

Code：

```python
if target_modules and target_modules == {"all"}:
    self.target_modules = auto_detect_lora_target_modules(self.base_model)
    self.target_modules.update(EMBEDDING_NAMES)
elif target_modules:
    self.target_modules = get_normalized_target_modules(target_modules)
else:
    self.target_modules = set()

if max_lora_rank is not None:
    self.max_lora_rank = max_lora_rank
else:
    self.max_lora_rank = max([x.r for x in self.configs.values()], default=0)

if self.lora_added_tokens_size is None:
    inferred_extra_vocab_size = next(
        (x.lora_added_tokens_size for x in self.configs.values() if x.lora_added_tokens_size > 0),
        0,
    )
    self.lora_added_tokens_size = inferred_extra_vocab_size
```

代码逻辑：
- 先建立 manager 的 target module 集合。
- 逐个 adapter config 校验或合并 target modules。
- 处理 DSA indexer fusion 与 LoRA target 的不兼容。
- rank 和 added token size 在末尾确定。

为什么这样写：
- shape 推断必须同时看 base model 和 adapter config，单靠一侧都不完整。
- rank 取最大值能让同一 pool 承载多个 rank 不同的 adapter。

不变量与失败模式：
- 显式 CLI target 存在时，adapter target 必须是其子集。
- 未识别的 target_modules 字符串会抛错，避免静默跳过权重。

Comment：
这段决定后面哪些模型层会被替换，以及每个 buffer 需要预留多大 rank。

### 2.2 init_memory_pool 与 set_lora_module 连接 buffer 和模型层

问题与约束：
- LoRA layer 需要从统一 memory pool 读取权重；但 base model 原始模块需要先被替换成带 LoRA 的 wrapper。

设计选择：
- `init_memory_pool` 创建全局 `LoRAMemoryPool`，`set_lora_module` 用 `get_lora_layer` 包装模块并通过 `replace_submodule` 原地替换。

Explain：
memory pool 拿到 base config、slot 上限、dtype、TP rank、target modules、eviction policy、MoE shared outer 和 strict/overlap loading 开关。创建后先 `fetch_new_loras({None})`，保证 base model 的 `None` 槽位存在。

来源：python/sglang/srt/lora/lora_manager.py L721-L746

Code：

```python
self.memory_pool = LoRAMemoryPool(
    self.base_hf_config,
    self.max_loras_per_batch,
    self.dtype,
    self.tp_size,
    self.tp_rank,
    self.max_lora_rank,
    self.target_modules,
    self.base_model,
    self.eviction_policy,
    self.lora_added_tokens_size,
    experts_shared_outer_loras=self.experts_shared_outer_loras,
    strict_loading=self.lora_strict_loading,
    enable_lora_overlap_loading=self.enable_lora_overlap_loading,
)
self.fetch_new_loras({None})
```

```python
def set_lora_module(self, module_name, module):
    lora_module = get_lora_layer(module, self.lora_backend)
    replace_submodule(self.base_model, module_name, lora_module)
    return lora_module
```

代码逻辑：
- memory pool 在 manager 中单例维护。
- `None` 作为 base model uid 参与槽位管理。
- module wrapper 由原模块类型和 backend 决定。

为什么这样写：
- forward 过程中 layer 只需要读预绑定的 buffer，不应关心 adapter 加载和淘汰逻辑。
- 把 base model 也放入 uid/slot 体系，能让无 LoRA 请求和 LoRA 请求共享 batch metadata。

不变量与失败模式：
- memory pool 必须在 `update_lora_info` 前创建。
- 被替换模块必须能被 `get_lora_layer` 支持，否则 target module 设置不成立。

Comment：
这段是模型结构和 LoRA buffer 的连接点。

### 2.3 init_lora_modules 处理 embedding、lm_head、普通线性层和 FusedMoE

问题与约束：
- 不同模型结构的 LoRA 注入点不一致：embedding/lm_head 有 vocab 分片问题，DeepSeek MLA 有 fused projection，MoE 需要同时处理 gate_up/down 和 expert routing。

设计选择：
- 遍历 `base_model.named_modules()`，按 target module 类型分别包装，并把每层 wrapped module 收集到 `self.lora_modules`。

Explain：
如果 `lm_head` 与 `embed_tokens` 共享权重，源码会显式构造一个共享 base weight 的 `ParallelLMHead`，让 `named_modules()` 能看到可包装的 lm_head。DeepSeek MLA 的 `fused_qkv_a_proj_with_mqa` 会设置 `first_output_dim` 边界。`FusedMoE` 只有在 target modules 同时包含 `gate_up_proj/down_proj` 时才被包装。

来源：python/sglang/srt/lora/lora_manager.py L748-L858

Code：

```python
self.lora_modules = [dict() for _ in range(self.base_hf_config.num_hidden_layers)]
self.embed_tokens_module = None
self.lm_head_module = None

if module_name == "embed_tokens" and "embed_tokens" in self.target_modules:
    self.embed_tokens_module = self.set_lora_module(module_name, module)
elif module_name == "lm_head" and "lm_head" in self.target_modules:
    self.lm_head_module = self.set_lora_module(module_name, module)
elif target_module in self.target_modules:
    lora_module = self.set_lora_module(module_name, module)
    self.lora_modules[layer_id][module_name] = lora_module
elif isinstance(module, FusedMoE) and all(
    target in self.target_modules for target in ["gate_up_proj", "down_proj"]
):
    lora_module = self.set_lora_module(module_name, module)
    self.lora_modules[layer_id][module_name] = lora_module
```

代码逻辑：
- 每个 hidden layer 有一个 module map。
- embedding 和 lm_head 单独保存。
- 普通 target module 按 suffix/name 匹配。
- FusedMoE 作为一个 wrapper 承载多个 expert LoRA target。

为什么这样写：
- LoRA 权重加载时需要按 layer_id 和 module_name 找到对应 wrapper 进行 TP/MoE slicing。
- 特殊层提前分流，避免用普通 Linear 规则处理 vocab 或 expert 维度。

不变量与失败模式：
- `get_layer_id` 必须能从 module name 解析 layer id。
- FusedMoE LoRA 要求 gate_up/down 同时启用，否则 expert 路径不完整。

Comment：
LoRA 注入不是简单替换所有 Linear，而是按模型结构拆成多类特殊路径。

### 2.4 LoRAAdapter 规范化 PEFT 权重名和 fused buffer 布局

### 2.5 Embedding LoRA 用 lookup 加 B 投影，不走普通 A matmul

问题与约束：
- embedding 的输入是 token id，不是 hidden state；普通 LoRA A 矩阵乘法不适用于词表查表路径。

设计选择：
- embedding wrapper 保存 `new_embeddings_buffer/embedding_A_buffer/embedding_B_buffer`，forward 时先做 LoRA A embedding lookup，再调用 backend 的 LoRA B sgemm。

Explain：
`set_lora_info` 只绑定 buffer 指针；`apply_lora` 读取 batch_info 中每个 token 对应的 adapter 槽位，先得到 rank 维的 A 输出，再用 B buffer 加到 base embedding output 上。

来源：python/sglang/srt/lora/layers.py L109-L140

Code：

```python
def set_lora_info(
    self,
    new_embeddings_buffer: Optional[torch.Tensor],
    embedding_A_buffer: torch.Tensor,
    embedding_B_buffer: torch.Tensor,
):
    self.set_lora = True
    self.new_embeddings_buffer = new_embeddings_buffer
    self.embedding_A_buffer = embedding_A_buffer
    self.embedding_B_buffer = embedding_B_buffer

    lora_a_output = self.run_lora_a_embedding(input_, batch_info)
    lora_output = self.lora_backend.run_lora_b_sgemm(
        x=lora_a_output,
        weights=self.embedding_B_buffer,
        output_offset=self.output_offset,
        output_offset_cpu=self.output_offset_cpu,
        base_output=base_output,
    )
    return lora_output
```

代码逻辑：
- buffer 绑定与 forward 执行分离。
- A 阶段按 token id lookup。
- B 阶段复用 backend 的 sgemm 路径。
- output offset 处理 batch 内 adapter 槽位偏移。

为什么这样写：
- embedding LoRA 的 A 权重语义是词表到 rank 的查表，不能按 hidden vector 矩阵乘。
- 仍复用 B sgemm，可以和普通 LoRA backend 共享 batch metadata。

不变量与失败模式：
- `input_` 必须是 token ids。
- embedding buffer 的 vocab 维要覆盖 base vocab 与允许的 added tokens。

Comment：
Embedding LoRA 是同一 adapter 槽位体系下的一条特殊 forward 路径。

## 3. Memory pool：槽位、换入和 batch metadata

### 3.1 LoRAMemoryPool 预分配所有目标模块的 A/B buffer

问题与约束：
- serving 过程中 adapter 会动态切换；如果每个 batch 都重新分配 GPU tensor，会破坏 CUDA graph 和吞吐稳定性。

设计选择：
- memory pool 按 `max_loras_per_batch` 和 `max_lora_rank` 预分配 buffer，并用 uid 与 buffer id 的双向映射管理槽位。

Explain：
构造函数保存 TP/MoE 上下文、eviction policy、A/B buffer dict、embedding/lm_head buffer 和 `uid_to_buffer_id/buffer_id_to_uid`。`can_support` 用 rank、added token size 和 target modules 子集判断 adapter 是否能进入当前 pool。

来源：python/sglang/srt/lora/mem_pool.py L132-L254

Code：

```python
self.A_buffer: Dict[str, List[torch.Tensor]] = {}
self.B_buffer: Dict[str, List[torch.Tensor]] = {}
self.embedding_A_buffer: Dict[str, torch.Tensor] = {}
self.embedding_B_buffer: Dict[str, torch.Tensor] = {}
self.lm_head_A_buffer: Dict[str, torch.Tensor] = {}
self.lm_head_B_buffer: Dict[str, torch.Tensor] = {}

self.buffer_id_to_uid: List[Union[str, None, EmptySlot]] = [
    EMPTY_SLOT
] * self.max_loras_per_batch

    if config.r > self.max_lora_rank:
        return False
    if config.lora_added_tokens_size > self.lora_added_tokens_size:
        return False
    target_module_names = get_normalized_target_modules(config.target_modules)
    if "all" in target_module_names:
        return True
    return target_module_names.issubset(self.target_modules)
```

代码逻辑：
- A/B buffer 以 target module 名为 key。
- embedding、lm_head 和 new embedding 使用独立 buffer。
- uid 到 slot、slot 到 uid 双向维护。
- `can_support` 在动态 load 前做快速兼容检查。

为什么这样写：
- 固定 buffer 让 forward 只更新索引和权重内容，不更新计算图结构。
- 双向映射让“请求 uid 找槽位”和“淘汰槽位找 uid”都能 O(1) 完成。

不变量与失败模式：
- adapter rank 不能超过 pool 预留 rank。
- target modules 超出 pool 支持集合时，即使权重能读到也不能加载。

Comment：
Memory pool 是 LoRA 高并发 serving 的核心状态容器。

### 3.2 prepare_lora_batch 优先空槽，再按策略淘汰非当前 batch 槽位

问题与约束：
- 当前 batch 需要的 adapter 必须都在 GPU buffer 中；但 full pool 下不能淘汰当前 batch 正在使用的 adapter，也不能淘汰 pinned adapter。

设计选择：
- `get_available_buffer_slot` 先找 `EMPTY_SLOT`，找不到再构造候选集合，跳过 `cur_uids` 和 pinned adapter，用 eviction policy 选 victim。

Explain：
候选中优先淘汰非 `None` adapter，只有当 batch 全是 LoRA 请求且没有其他 adapter 可淘汰时才会淘汰 base model 的 `None` 槽。选中 victim 后同时更新 uid map、eviction policy 和 slot 状态。

来源：python/sglang/srt/lora/mem_pool.py L674-L754

Code：

```python
for buffer_id in range(self.max_loras_per_batch):
    if self.buffer_id_to_uid[buffer_id] == EMPTY_SLOT:
        return buffer_id

        continue

        if lora_ref and lora_ref.pinned:
            continue

self.eviction_policy.remove(victim_uid)
self.buffer_id_to_uid[victim_buffer_id] = EMPTY_SLOT
```

代码逻辑：
- 当前 batch 的所有 uid 先 mark used。
- 缺失 uid 逐个申请 slot。
- 空 slot 无需 eviction。
- 满 slot 时只从安全候选中选 victim。

为什么这样写：
- 当前 batch adapter 被淘汰会导致本轮 forward 读错权重。
- pinned adapter 表示用户要求常驻，不能被普通 eviction 破坏。
- `None` 代表 base model，优先保留可让混合 batch 继续服务无 LoRA 请求。

不变量与失败模式：
- 如果没有可淘汰候选，会抛出 `ValueError`，提示 pinned/active LoRA 数过多。
- `cur_uids` 数量必须不超过 `max_loras_per_batch`，上层 manager 已 assert。

Comment：
槽位管理的本质是把动态 adapter 集合压进固定大小 GPU buffer。

### 3.3 load_lora_weight_to_buffer 先校验权重名，缺失权重写零

问题与约束：
- adapter 可能缺少某个 target module 权重，或者包含当前模型不支持的权重名；如果直接覆盖部分 buffer，可能留下被淘汰 adapter 的残留权重。

设计选择：
- 加载前统计匹配和跳过的权重；strict 模式下跳过权重报错。实际写入时，缺失权重对应的 buffer view 统一 zero。

Explain：
`load_lora_weight_tensor` 对 `None` 权重调用 `zero_()`，否则 assert buffer view 与 weight shape 完全一致，再复制权重。`uid is None` 路径专门清空 A buffer，用于 base model 槽位。

来源：python/sglang/srt/lora/mem_pool.py L764-L826

Code：

```python

代码逻辑：
- 内部 helper 统一处理 copy 和 zero。
- base model 槽位只需要清空 LoRA A 路径。
- adapter 权重名先映射到 target module。
- strict loading 决定跳过权重是 warning 还是 error。

为什么这样写：
- LoRA delta 缺失应等价于零增量，而不是继承旧 adapter 残留。
- shape assert 在 copy 前触发，避免部分写入造成难以定位的运行时错误。

不变量与失败模式：
- buffer view 与权重 shape 必须完全一致。
- strict loading 开启后，任何不匹配 target module 的权重都会让加载失败。

Comment：
这段保证 adapter 换入是干净覆盖，而不是增量覆盖。

### 3.7 CUDA graph 与 decode sgemm routing 使用预分配 metadata

问题与约束：

设计选择：

Explain：

    bs=max_bs_in_cuda_graph,
    use_cuda_graph=True,
    seg_lens=torch.full((max_bs_in_cuda_graph,), num_tokens_per_bs, dtype=torch.int32),
    seg_indptr=torch.zeros(max_bs_in_cuda_graph + 1, dtype=torch.int32),
    weight_indices=torch.zeros(max_bs_in_cuda_graph, dtype=torch.int32),
    lora_ranks=torch.zeros(mlpb, dtype=torch.int32),
    scalings=torch.zeros(mlpb, dtype=torch.float),
    permutation=None,
)

perm = torch.argsort(wi, stable=True).to(torch.int32)
sorted_wi = wi[perm]
adapter_ids = torch.arange(mlpb, device=wi.device, dtype=torch.int32)
seg_starts = torch.searchsorted(sorted_wi, adapter_ids)
seg_ends = torch.searchsorted(sorted_wi, adapter_ids, right=True)
seg_lens = seg_ends - seg_starts
```

代码逻辑：
- graph metadata tensor 在初始化阶段创建。
- request-level batch info 和 sgemm-level batch info 分开。
- decode routing 按 slot 排序，得到 adapter 连续段。
- graph 分支原地更新预分配 tensor。

为什么这样写：
- CUDA graph 需要形状和对象稳定。
- 按 adapter 合并后，每次 kernel launch 处理更规整的 segment，减少散乱访问。

不变量与失败模式：
- graph batch size 不能超过初始化时的 `max_bs_in_cuda_graph`。
- `weight_indices` 必须已经拷到 device，`argsort/searchsorted` 才能正确构造 routing。

Comment：
LoRA 的 graph 支持不是只固定计算图，还要固定 batch metadata 的存储位置。

### 3.8 backend prepare_lora_batch 把 CPU metadata 异步拷到 device

问题与约束：
- manager 生成的是 Python list；kernel 需要 device tensor。同步拷贝或 D2H 查询会影响 decode 路径延迟。

设计选择：
- backend 先把 list 构造成 pinned CPU tensor，再按 CUDA graph 或非 graph 路径写入 `LoRABatchInfo`，最后异步 copy 到 device。

Explain：
extend 模式的 segment lens 来自 `forward_batch.extend_seq_lens`，decode 模式每个请求长度为 1。metadata 拷贝后，backend 补充 MoE LoRA info，decode 模式再计算 sgemm routing，lm_head prune 场景则额外预计算 lm_head batch info。

来源：python/sglang/srt/lora/backend/triton_backend.py L229-L312

Code：

```python
weight_indices_tensor = torch.tensor(
    weight_indices, dtype=torch.int32, pin_memory=True, device="cpu"
)
lora_ranks_tensor = torch.tensor(
    lora_ranks, dtype=torch.int32, pin_memory=True, device="cpu"
)
scalings_tensor = torch.tensor(
    scalings, dtype=torch.float, pin_memory=True, device="cpu"
)

batch_info.lora_ranks[: self.max_loras_per_batch].copy_(
    lora_ranks_tensor, non_blocking=True
)
batch_info.scalings[: self.max_loras_per_batch].copy_(
    scalings_tensor, non_blocking=True
)
batch_info.weight_indices[:bs].copy_(weight_indices_tensor, non_blocking=True)

batch_info = self._add_moe_lora_info(forward_batch, batch_info)
self.batch_info = batch_info
```

代码逻辑：
- list 先转 pinned CPU tensor。
- graph 路径复用已有 batch_info。
- 非 graph 路径按当前 batch 创建 segment tensor。
- rank、scaling、weight index 非阻塞拷贝到 device。

为什么这样写：
- pinned memory + non-blocking copy 降低 host 到 device metadata 更新成本。
- graph 分支原地更新，避免 capture 后对象变化。

不变量与失败模式：
- CUDA graph 路径必须已经初始化 `cuda_graph_batch_info`。
- `forward_batch.batch_size` 必须不超过对应 metadata tensor 容量。

Comment：
这段是每个 forward batch 进入 LoRA kernel 前的最后一道 metadata 准备。

### 3.9 Manager 侧 CUDA graph 初始化分两阶段

问题与约束：
- MoE LoRA buffer 需要在 memory profiling 前分配；dense batch metadata 又依赖 memory pool 初始化后的 graph runner 阶段。

设计选择：
- `init_cuda_graph_moe_buffers` 作为 Phase 1 提前初始化 MoE intermediate buffer；`init_cuda_graph_batch_info` 作为 Phase 2 初始化 dense LoRA batch metadata，并可预创建 two-stream 资源。

Explain：
源码 docstring 明确两阶段调用位置：MoE buffer 在 `init_memory_pool()` 前，让 profiler 计入显存；dense metadata 在 `CudaGraphRunner.__init__()` 中初始化。experimental overlap loading 的 stream/event 也在 Phase 2 创建，避免 graph capture 内分配。

来源：python/sglang/srt/lora/lora_manager.py L114-L152

Code：

```python
def init_cuda_graph_batch_info(
    self, max_bs_in_cuda_graph: int, num_tokens_per_bs: int
):
    self.max_bs_in_cuda_graph = max_bs_in_cuda_graph
    self.lora_backend.init_cuda_graph_batch_info(
        max_bs_in_cuda_graph=max_bs_in_cuda_graph,
        num_tokens_per_bs=num_tokens_per_bs,
    )
    if _SGLANG_EXPERIMENTAL_LORA_OPTI and self.enable_lora_overlap_loading:
        self.memory_pool.init_two_stream_overlap()

def init_cuda_graph_moe_buffers(
    self, max_bs: int, max_loras: int, compute_dtype, moe_layer
):
    self.lora_backend.init_cuda_graph_moe_buffers(
        max_bs=max_bs,
        max_loras=max_loras,
        compute_dtype=compute_dtype,
        moe_layer=moe_layer,
    )
```

代码逻辑：
- Phase 2 保存 graph 最大 batch size。
- backend 负责具体 batch metadata tensor 初始化。
- Phase 1 只处理 MoE graph buffer。
- overlap loading 的 stream 资源在 graph capture 外创建。

为什么这样写：
- 显存 profiling 必须看见 MoE LoRA 额外 buffer，否则后续可能 OOM。
- CUDA graph capture 不应包含动态 stream/event 分配。

不变量与失败模式：
- 调用顺序错误会让 profiler 漏算或 graph capture 失败。
- two-stream overlap 只在实验开关和 overlap loading 同时开启时初始化。

Comment：
LoRA 的 CUDA graph 支持跨 manager、backend 和 memory pool 多处协同。

## 4. Kernel 与调度公平性

### 4.1 chunked_sgmv_lora_shrink_forward 按 segment 调 LoRA A shrink kernel

问题与约束：
- 一个 batch 里不同 token/request 可能使用不同 adapter；LoRA A shrink 需要按 segment 读取不同 slot 的权重，同时支持 qkv/gate_up 等多 slice rank 输出。

设计选择：
- CSGMV wrapper 从 `LoRABatchInfo` 读取 segment indptr、weight indices、rank 和 permutation，按 autotune config 选择 block shape 后 launch Triton kernel。

Explain：
`num_slices` 表示 fused 输出的 slice 数，qkv 为 3、gate_up 为 2、普通层为 1。CUDA graph 下 grid 的 segment 维使用 `weight_indices.shape[0]` 保持稳定，非 graph 下使用实际 `num_segments`。

来源：python/sglang/srt/lora/triton_ops/chunked_sgmv_shrink.py L125-L196

Code：

```python
BLOCK_M = batch_info.max_len
MAX_RANK = weights.shape[1] // num_slices
config = get_lora_shrink_config(
    K=weights.shape[2], R=MAX_RANK, num_slices=num_slices, chunk_size=BLOCK_M
)

num_segments = batch_info.num_segments
segment_grid = (
    batch_info.weight_indices.shape[0]
    if batch_info.use_cuda_graph
    else num_segments
)
grid = (triton.cdiv(N, BLOCK_N), segment_grid)

_chunked_lora_shrink_kernel[grid](
    x=x,
    weights=weights,
    output=output,
    seg_indptr=batch_info.seg_indptr,
    weight_indices=batch_info.weight_indices,
    lora_ranks=batch_info.lora_ranks,
    permutation=batch_info.permutation,
    num_segs=segment_grid,
    N=N,
    K=K,
    NUM_SLICES=num_slices,
)
```

代码逻辑：
- 输入和权重必须 contiguous。
- block config 由 K、rank、slice 数和 chunk size 决定。
- grid 的第二维是 segment 数。
- kernel 通过 batch_info 查每段的 adapter slot 和 rank。

为什么这样写：
- segmented GEMV 能在同一次 launch 中处理多 adapter batch。
- graph 路径固定 grid 维度，满足 capture/replay 的形状稳定性。

不变量与失败模式：
- `weights.shape[1]` 必须能被 `num_slices` 整除。
- `batch_info.seg_indptr/weight_indices/lora_ranks` 必须与当前 batch 和 pool slot 对齐。

Comment：
CSGMV 是 LoRA dense 路径把“多 adapter”压进一个 kernel 调度的关键实现。

### 4.2 LoRADrainer 统计等待与运行请求，识别饥饿 adapter

问题与约束：
- 如果 running batch 长期占满 LoRA slot，等待队列中的 adapter 可能一直无法进入 batch；但直接抢占运行请求又会破坏推理。

设计选择：
- drainer 每轮调度统计每个 adapter 的等待请求数、最长等待时间和运行请求剩余 token，再按饥饿阈值选择要 draining 的运行 adapter。

Explain：
`_update_adapter_stats` 先清零旧统计，再扫 waiting queue 更新等待数量与最大等待时间，扫 running requests 更新最大剩余生成 token。后续选择 draining victim 时，会偏向剩余 token 更少的 running adapter。

来源：python/sglang/srt/lora/lora_drainer.py L76-L99

Code：

```python
for stats in self.adapter_to_stats.values():
    stats._reset_stats()

for req in waiting_queue:
    stats = self.adapter_to_stats[req.lora_id]
    stats.num_waiting_reqs += 1
    stats.max_wait_time_secs = max(
        stats.max_wait_time_secs,
        time.monotonic() - req.time_stats.wait_queue_entry_time,
    )

for req in running_reqs:
    stats = self.adapter_to_stats[req.lora_id]
    stats.max_remaining_tokens = max(
        stats.max_remaining_tokens,
        req.sampling_params.max_new_tokens - len(req.output_ids),
    )
```

代码逻辑：
- 每轮调度重新计算统计。
- waiting queue 贡献等待时间和数量。
- running requests 贡献剩余 token 上界。
- 统计按 `lora_id` 聚合。

为什么这样写：
- 饥饿判断需要等待时间；drain 成本需要估算当前运行 adapter 还要占用 slot 多久。
- 用 max remaining tokens 保守估计一个 adapter 完全释放 slot 前的最长等待。

不变量与失败模式：
- 请求必须有稳定的 `lora_id` 和 wait queue entry time。
- max_new_tokens 与已输出 token 的差值只是上界，实际结束可能更早。

Comment：
Drainer 不做强制中断，它只为后续调度提供“不要再给谁追加请求”的信号。

### 4.3 _update_draining_loras 为饥饿 adapter 腾出 slot

问题与约束：
- 当 running adapter 数已经达到 `max_loras_per_batch`，新 adapter 无法进入；需要选择某些正在运行的 adapter 停止接收新请求。

设计选择：
- 找到等待时间超过阈值的 starving adapters，按等待时间排序；对每个 starving adapter，选择一个尚未 draining 且剩余 token 最少的 running adapter 标记为 `is_draining_for`。

Explain：
如果当前 running adapter 数少于 slot 上限，说明还有可用 LoRA slot，不需要 draining。已有 draining 目标不会重复分配。victim 选择最小 `max_remaining_tokens`，让 slot 尽快自然释放。

来源：python/sglang/srt/lora/lora_drainer.py L101-L158

Code：

```python
running_adapter_ids = {req.lora_id for req in running_reqs}
if len(running_adapter_ids) < self.max_loras_per_batch:
    return None

new_starving_adapters = starving_adapters - draining_for_adapters
if not new_starving_adapters:
    return None

eligible_to_drain_adapters = {
    adapter
    for adapter in running_adapter_ids
    if self.adapter_to_stats[adapter].is_draining_for is None
}

for starving_adapter in sorted_new_starving_adapters:
    min_eligible_adapter = min(
        eligible_to_drain_adapters,
        key=lambda adapter_id: self.adapter_to_stats[adapter_id].max_remaining_tokens,
    )
    self.adapter_to_stats[min_eligible_adapter].is_draining_for = starving_adapter
    eligible_to_drain_adapters.remove(min_eligible_adapter)
```

代码逻辑：
- running adapter 未占满槽位时直接返回。
- starving adapter 去掉已经被服务的 draining target。
- victim 只能从未处于 draining 的 running adapter 里选。
- 每个 victim 只服务一个 starving adapter。

为什么这样写：
- draining 是协作式释放，不打断已有请求，只阻止继续扩张。
- 选择剩余 token 少的 adapter，能更快让等待 adapter 获得 slot。

不变量与失败模式：
- `max_wait_time_secs` 过小会频繁触发 draining，影响稳定吞吐。
- 过大则可能让低频 adapter 长时间等待。

Comment：
这是 LoRA 多租户调度的公平性补丁：牺牲一点局部吞吐，换等待 adapter 的进入机会。

### 4.4 can_schedule 在 draining 期间限制新请求进入

问题与约束：
- 标记 draining 后，如果仍持续给该 adapter 加入长请求，slot 永远不会释放。

设计选择：
- `can_schedule` 对非 draining adapter 直接放行；对 draining adapter，只允许 max_new_tokens 不超过当前剩余 token 上界乘 tolerance 的请求。

Explain：
当 adapter 已经不在 running set 中时，`_update_fully_drained_loras` 会清除 draining 标记。调度限制不是绝对禁止所有请求，而是只允许不会显著延长 draining 时间的小请求。

来源：python/sglang/srt/lora/lora_drainer.py L160-L191

Code：

```python
running_adapter_ids = {req.lora_id for req in running_reqs}
for adapter_id, stats in self.adapter_to_stats.items():
    if stats.is_draining_for is None:
        continue
    if adapter_id not in running_adapter_ids:
        stats.is_draining_for = None

def can_schedule(self, req: Req) -> bool:
    stats = self.adapter_to_stats[req.lora_id]
    if not stats.is_draining_for:
        return True

    return (
        req.sampling_params.max_new_tokens
        <= stats.max_remaining_tokens * DRAIN_SCHEDULE_TOLERANCE
    )
```

代码逻辑：
- draining adapter 没有运行请求后清除状态。
- 非 draining 请求不受限制。
- draining 请求按长度阈值判断是否可追加。

为什么这样写：
- 完全禁止可能降低短请求吞吐；完全放行会让 draining 无效。
- tolerance 给短请求留余地，减少调度抖动。

不变量与失败模式：
- `max_remaining_tokens` 必须持续更新，否则 schedule 判断会过时。
- tolerance 过大可能拖慢 slot 释放。

Comment：
Drainer 的实际效果依赖调度器在入队/选 batch 时调用 `can_schedule`。
